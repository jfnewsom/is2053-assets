Lab Submission Format
What a clean lab-X-Y.py looks like
Overview

Every Python file you submit to CodeGrade follows the same structure. The header at the top of the file, the indentation inside functions, the blank lines between sections, and the inline comments next to your code are all checked automatically. Getting these right is worth up to 23 points per lab (12 Code Style + 11 Documentation) on the grading rubric.

This page shows you exactly what a clean submission looks like. Save it. Open it next to your editor when you write your first few labs. After a couple of weeks the format becomes muscle memory.

The Format at a Glance

Open the image in a new tab for a closer look. The colors on the image match the four rule categories below.

Required Structural Elements

The green items in the image. These are checked by the Documentation rubric.

Element What it looks like
Docstring header A triple-quoted block at the very top of the file containing the filename, assignment ID, project line, your name, your UTSA ID (abc123), your section, and a one-line description.
Named constants UPPER_SNAKE_CASE names defined at the top of the module, above any function. Each gets a one-line comment explaining what it represents.
Function docstring A triple-quoted string on the first line inside every function you write, briefly explaining what the function does.
Inline comments A short # comment immediately above each meaningful block of code, in plain English. Minimum three inline comments per file.
Standard entry point The two-line if __name__ == '__main__': guard at the bottom of the file that calls main(). Always the last thing in the file.
Blank-Line Rules

The red zones in the image. These are PEP 8 spacing rules enforced by Flake8.

Where How many blank lines
Before def main(): Two blank lines separate the module-level code from the function definition.
Between code blocks inside a function One blank line separates logical sections of code inside main().
After main() ends Two blank lines separate the function from the if __name__ guard.
At the end of the file Exactly one newline — no extra blank lines after main() is called.
Indentation Rule

The blue column in the image.

subdirectory_arrow_right
Every colon (:) starts an indented block

This is a general Python rule, not just a style preference. After a def, if, elif, else, for, while, or class line ending in a colon, the very next line must be indented by exactly four spaces.

VS Code handles this for you automatically. When you press Enter after a line ending in a colon, VS Code auto-indents the next line. When you press the Tab key inside an already-indented block, VS Code inserts 4 spaces (not a literal tab character) for Python files. You shouldn't have to count spaces manually.

Flake8 flags incorrect indentation with codes E111 (indentation not a multiple of 4) and E117 (over-indented).

Line Length

The amber ruler in the image.

Python lines should be at most 79 characters wide. This is the PEP 8 standard and is enforced by Flake8 rule E501. If a line gets too long, break it across multiple lines using parentheses, brackets, or backslash continuation. Most of your lab code will be well under this limit.

How This Gets Graded

Following this format earns you points in two of the five CodeGrade rubric categories:

  • Documentation (11 points) — presence check: header docstring, function docstrings, and at least three inline comments. Either you have them or you don't.
  • Code Style (12 points) — Flake8 violation count. Zero violations gets the full points; the score ladder is in the Flake8 Guide.

The other three rubric categories (Scope Compliance, Correctness, Required Techniques) check what your code does, not how it's formatted. See Grading Info for the full point breakdown.

Common Mistakes
What students do What goes wrong
Use # Name: hash comments instead of a docstring at the top Documentation check fails. The header must be inside a triple-quoted """ docstring at the top of the file.
Tab characters in the file (often from pasted code) Flake8 flags it (W191 / E101) and Python may raise TabError. VS Code defaults to inserting 4 spaces when you press Tab in a Python file, so this is almost always from pasting code from another source. Use VS Code's Convert Indentation to Spaces command (Command Palette → type "convert") to clean it up.
Skip the function docstring Lose Documentation points. Every function you write needs one short triple-quoted line right after the def.
Only one or two inline comments The check requires at least three inline comments. Add comments before each meaningful section of code.
One blank line before def main(): instead of two Flake8 E302. Always two blank lines before and after top-level function definitions.
Extra blank lines at the end of the file Flake8 W391. The file should end with exactly one newline after main().
