5
Disable AI Autocomplete in VS Code
3 minutes — Critical for new programmers

VS Code now ships with AI-powered code suggestions turned on by default. It sounds helpful, but it will hurt you in this course. The AI doesn’t know what chapter we’re on, so it happily suggests list comprehensions, f-strings, and other techniques we haven’t covered yet. CodeGrade flags those as scope violations and auto-zeroes the assignment. The suggestions are also frequently just wrong for your specific problem.

Turn off all three of these in VS Code Settings (press Cmd + , on Mac or Ctrl + , on Windows):

  1. Search editor.inlineSuggest.enabled — uncheck the box. This kills inline AI completions.
  2. Search github.copilot.enable — set to false, or uninstall the GitHub Copilot extension entirely.
  3. Search IntelliCode — disable or uninstall both the “IntelliCode” and “IntelliCode API Usage Examples” extensions.
check_circle
How to Verify It Worked

After making these changes, when you type code you should only see basic autocomplete (Python keywords like if, while, print, and variable names you’ve already typed). You should not see entire lines of suggested code appearing in gray. Those gray ghost-lines are the AI—if you see them, you missed a setting.

I can’t enforce this from my end. But students who lean on AI suggestions in the early modules consistently struggle harder later, because they never built the foundation. See the full AI Policy for the longer version of why.

tune
Each Folder Is Pre-Configured for Python

Every assignment folder in the zip ships with a hidden .vscode/ sub-folder containing workspace settings tuned for this course. When you open one of these folders in VS Code, you’ll automatically see:

  • Three faint vertical lines in the editor — at columns 4, 8, and 79. The first two are visual guides for your first and second indent levels. The third is the PEP 8 line-length limit — Flake8 will flag any line that crosses it.
  • Tab key inserts 4 spaces, not a tab character. Python style requires spaces, and CodeGrade will reject mixed indentation.

You don’t need to do anything to enable this — it just works once you open the folder. If you’re curious, peek inside any .vscode/settings.json to see what’s configured.

error
The One Rule That Makes Everything Easier

Always “File → Open Folder” on the specific assignment folder you’re working in — never the parent is2053/ or code/ folder.

Why? When you open the assignment folder directly, Python automatically looks in that folder for data files like hazards.txt or city_info.txt. Open the parent folder instead, and Python looks in the parent — the data files are nowhere to be found and you get FileNotFoundError errors that make no sense.

Example for Lab 4.3: open is2053/code/modules/mod4/lab-4-3/ as the workspace, not is2053/. Drag the assignment folder onto the VS Code icon, or use File → Open Folder and pick it.

lightbulb
Why So Many Folders?

Modules 1–3 keep things simple — each lab is a single .py file in its own folder. Once you hit Module 4, your labs add a function_library.py module plus data files (city_distances.txt, hazards.txt, trivia.txt, etc.). Module 5 introduces multi-file class projects. Per-assignment folders isolate each lab’s data and prevent old files from leaking into new work.

Hello World — Your First Submission
15–20 minutes — Write a Python program, submit it to CodeGrade, get the painful first time out of the way

You’re going to do a complete end-to-end workflow for the first time: open a folder in VS Code, write a Python program, run it locally, then submit it to CodeGrade (the autograder we use for every coding assignment in this course). This submission is worth zero points — its only job is to make sure your setup works and to walk you through CodeGrade signup before there’s a grade on the line.

Step 1 — Open Folder, Not File (Your Day-One Workflow)

Right now you have two ways to get hello.py in front of VS Code. Pay attention to which one we’re using, because this is the workflow rule every assignment in this course depends on.

  • The wrong way — double-click hello.py in Finder/Explorer. VS Code opens with just that one file, disconnected from any folder. Python doesn’t know what directory to look in for data files. You won’t notice the problem today (there are no data files yet), but starting in Module 4 you’ll get FileNotFoundError messages that make no sense.
  • The right way — open the folder in VS Code first, then click the file from the sidebar. Python automatically uses that folder as its working directory. Your data files just work. This is how every professional Python developer works.

We’re going to do it the right way every single time, starting now. Even though hello.py has no data files and you could get away with the wrong way today, building the habit on day one means you never trip over the problem later.

To open the folder:

  1. In Finder/Explorer, navigate to is2053/start_here/
  2. Drag the start_here folder onto the VS Code icon — OR — open VS Code and use File → Open Folder, then pick start_here
  3. You should now see hello.py listed in the Explorer sidebar on the left. This means VS Code has the folder open as a workspace, and the file is accessible from inside it. That’s the workflow.
Step 2 — Write the program and run it locally

Now that the folder is open in VS Code, click hello.py in the Explorer sidebar on the left. It opens in the editor. Type exactly this line: 

print("Hello, World!")
match_case
Match the Cases and Punctuation Exactly

CodeGrade is going to compare what your program prints against what it expects, character by character. It is case-sensitive and punctuation-sensitive.

Type the line exactly as shown above:

  • Capital H in Hello, capital W in World
  • A comma after Hello, followed by a single space
  • An exclamation point at the end
  • Double quotes around the whole string

If you type print("hello world") or print("Hello world!") or even print("Hello,World!") (missing the space), the test will fail. Get used to this now: every assignment in this course is graded by a computer that does not interpret — only compare. Exact matters.

The easiest way to nail this: copy the spec, don’t retype it. When the instructions show you the exact text the program should print, select it, copy it, paste it into your code. Zero typos, zero case mistakes, zero missed punctuation. Professionals do this constantly — nobody manually retypes a string when they can copy it from the spec. Save your typing energy for the parts of the program that actually require thinking.

“Here’s a thing they don’t teach you in school but you learn fast in industry: the spec is the contract. When a client signs off on a build spec that says the output reads ‘Hello, World!’, that’s the deliverable. Not ‘hello, world’, not ‘Hello World’. Your job is to execute the spec, not relitigate it. The spec was decided above your pay grade.

If you think there’s a better way to do it, great — say so. Bring it up. Push back. Open the conversation. But until the spec gets formally changed, you build what the contract says. That’s the difference between a junior dev and a developer the client actually trusts with the keys.”

Danny Cavanaugh
Lead Developer, Bat City Collective

Save the file (Cmd + S on Mac, Ctrl + S on Windows). Then click the Play button in the upper-right corner of the editor. A terminal panel opens at the bottom of VS Code and you should see: 

Hello, World!

That’s your first Python program working locally. Now we send it to CodeGrade.

error
If the Play Button Doesn’t Work

The most common cause: Python wasn’t added to PATH during install. If you see “Python is not recognized” or a similar error, here’s the fix:

On Windows: Re-run the Python installer from python.org and this time check the “Add Python to PATH” box at the bottom of the first installer screen before you click Install. It takes 2 minutes and is the simplest fix. If you already know how to edit the PATH environment variable in Windows, you can do that instead. If you just said “what is the PATH?” — just reinstall.

On Mac: VS Code probably hasn’t picked which Python to use yet. Open the Command Palette with Cmd + Shift + P, type Python: Select Interpreter, and pick the one with the highest version number (should be 3.14 or newer).

After either fix, close VS Code completely, reopen it, open the start_here folder again, and try the Play button. Still stuck? Drop a note in Discord #help-python or book a 1-on-1.
