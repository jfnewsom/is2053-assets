5
Disable AI Autocomplete in VS Code
3 minutes — Critical for new programmers

VS Code now ships with AI-powered code suggestions turned on by default. It sounds helpful, but it will hurt you in this course. The AI doesn’t know what chapter we’re on, so it happily suggests list comprehensions, f-strings, and other techniques we haven’t covered yet. CodeGrade flags those as scope violations and auto-zeroes the assignment. The suggestions are also frequently just wrong for your specific problem.

You’ll work in two different panels in VS Code for this checkpoint — first Settings, then Extensions. The keyboard shortcuts and search bars are different in each, so the steps below tell you which panel to switch to before each search.

Step 1 — In the Settings panel

Open Settings with Cmd + , (Mac) or Ctrl + , (Windows). Use the search bar at the top of the Settings page:

  1. Search editor.inlineSuggest.enabled and uncheck the box. This kills the gray ghost-text predictions that appear inline as you type.
Step 2 — In the Extensions panel

Now switch to Extensions with Cmd + Shift + X (Mac) or Ctrl + Shift + X (Windows). The search bar at the top of the Extensions sidebar is a different search bar from the one in Settings — it searches installed and available extensions, not settings keys.

  1. Search GitHub Copilot. If you see it in your installed list, click the gear icon and choose Uninstall. (If it’s not installed, skip this one — nothing to do.)
  2. Search GitHub Copilot Chat. This is the tricky one. New VS Code installs ship Copilot Chat as a built-in extension, which means there’s no Uninstall button — only a dropdown labeled “Disable AI Features”. Click the dropdown and choose Disable AI Features. This is the step that turns off the right-side CHAT panel and the “Generate code” inline prompt — both of those come from Copilot Chat, not from the inline suggestion setting in Step 1.
  3. Search IntelliCode. Disable (or uninstall) both IntelliCode and IntelliCode API Usage Examples using the gear icon on each.
check_circle
How to Verify It Worked

After making all the changes above, you should see no AI surfaces anywhere in VS Code. Specifically, all three of these things should be gone:

  • No gray ghost-text appearing inline as you type. Basic autocomplete is fine and expected — Python keywords like if, while, print and variable names you’ve already typed will still pop up in a suggestion dropdown. What should be gone is entire lines of suggested code appearing in gray ahead of your cursor. If those are still there, the editor.inlineSuggest.enabled setting didn’t take — go back and re-check Step 1.
  • No “Generate code” inline prompt floating in the editor near your cursor. That prompt comes from Copilot Chat, not from the inline suggestion setting. If you still see it, Copilot Chat’s AI features are still enabled.
  • No CHAT panel on the right side of the VS Code window. Same source — the right-side CHAT panel disappears once Copilot Chat’s AI features are disabled.

If any of these are still showing up, go back to the Extensions panel and confirm GitHub Copilot Chat reads “AI Features Disabled” next to its name.

I can’t enforce this from my end. But students who lean on AI suggestions in the early modules consistently struggle harder later, because they never built the foundation. See the full AI Policy for the longer version of why.

Set Up Your Workspace Folder
5 minutes — The folder structure that keeps you sane

You’re going to write a lot of code this semester. One folder per assignment is the simplest way to keep work organized, recover when something breaks, and avoid accidentally overwriting your previous lab.

Keeping a code base organized is a real skill — one of the unglamorous habits that separates developers who ship clean work from ones who lose half a day every week to “where did I put that?” Never organized a code base before? Don’t worry about it. I’ve set up the folder structure for you in the starter zip below, along with empty .py file templates pre-named exactly what CodeGrade expects. Just download it, drop it in your Documents folder, open the assignment folder when it’s time to work, and fill in the file. The organization is handled.

Download and unzip the starter folder:

Unzip it into your Documents folder, or anywhere easy to find. One small note: cloud-synced folders (OneDrive, Dropbox, iCloud Drive) can occasionally cause file-locking issues when VS Code, Python, and the cloud syncer all try to touch the same file at once. Most students never see this — I sync my own course code through OneDrive without issue — but if you ever get a mysterious “file is locked” or “permission denied” error mid-edit, cloud sync is the first thing to check.

Each Folder Is Pre-Configured for Python

Every assignment folder in the zip ships with a hidden .vscode/ sub-folder containing workspace settings tuned for this course. When you open one of these folders in VS Code, you’ll automatically see:

  • Three faint vertical lines in the editor — at columns 4, 8, and 79. The first two are visual guides for your first and second indent levels. The third is the PEP 8 line-length limit — Flake8 will flag any line that crosses it.
  • Tab key inserts 4 spaces, not a tab character. Python style requires spaces, and CodeGrade will reject mixed indentation.
  • Files save with LF line endings, not Windows-style CRLF. CodeGrade runs on Linux and expects LF — using the zip means you never have to think about this.
  • The Play button always runs your file, instead of dumping you into a >>> prompt. (Newer versions of the Python extension have a bug where the Play button opens an interactive shell instead of executing the script. The starter workspace disables the setting that causes this.)

You don’t need to do anything to enable any of this — it just works once you open the folder. If you’re curious, peek inside any .vscode/settings.json to see what’s configured.

build
Building your own folder? Click for line endings note

If you opted out of the starter zip and built your own folder structure, you’ll want to know about line endings — one of the silent ways CodeGrade can reject code that runs fine on your machine.

The problem: Windows saves files with CRLF line endings (carriage return + line feed). CodeGrade runs on Linux, which expects LF only. That extra carriage return character can garble your output to the autograder even though it looks fine in VS Code.

The symptom: Your code prints the expected output locally, but CodeGrade says the output is wrong — especially when the expected output spans multiple lines and the autograder reports it as one mashed-up line.

The 5-second fix:

  1. Open your .py file in VS Code
  2. Look at the bottom-right corner of the window — you’ll see CRLF
  3. Click on CRLF
  4. Select LF from the dropdown
  5. Save the file
  6. Resubmit to CodeGrade

The permanent fix: Open VS Code Settings (Cmd + , on Mac, Ctrl + , on Windows), search for eol, and change Files: Eol to \n (LF). Every new file you create will use LF from now on. (This is exactly what the starter zip’s .vscode/settings.json does — if you used the zip, you already have this and can ignore this entire section.)

If your code runs correctly on your machine but CodeGrade gives you weird output errors, check your line endings before you panic.

build
Building your own folder? Click for Play-button-shows->>> note

Recent versions of VS Code’s Python extension have a bug where clicking the Play button on a .py file opens an interactive Python shell (a >>> prompt) instead of running your script. Frustrating because it looks like the Play button is doing nothing useful.

The fix: In VS Code Settings (Cmd + , on Mac, Ctrl + , on Windows), search for REPL Smart Send, and uncheck Python › REPL: Enable REPL Smart Send. Restart VS Code. The Play button will now run your file like normal.

If you used the starter zip, the workspace already disables this setting and you’ll never see the bug.

The One Rule That Makes Everything Easier

Always “File → Open Folder” on the specific assignment folder you’re working in — never the parent is2053/ or code/ folder.

Why? When you open the assignment folder directly, Python automatically looks in that folder for data files like hazards.txt or city_info.txt. Open the parent folder instead, and Python looks in the parent — the data files are nowhere to be found and you get FileNotFoundError errors that make no sense.

Example for Lab 4.3: open is2053/code/modules/mod4/lab-4-3/ as the workspace, not is2053/. Drag the assignment folder onto the VS Code icon, or use File → Open Folder and pick it.

Modules 1–3 keep things simple — each lab is a single .py file in its own folder. Once you hit Module 4, your labs add a function_library.py module plus data files (city_distances.txt, hazards.txt, trivia.txt, etc.). Module 5 introduces multi-file class projects. Per-assignment folders isolate each lab’s data and prevent old files from leaking into new work.

15–20 minutes — Write a Python program, submit it to CodeGrade, get the painful first time out of the way

You’re going to do a complete end-to-end workflow for the first time: open a folder in VS Code, write a Python program, run it locally, then submit it to CodeGrade (the autograder we use for every coding assignment in this course). This submission is worth zero points — its only job is to make sure your setup works and to walk you through CodeGrade signup before there’s a grade on the line.

Step 1 — Open Folder, Not File (Your Day-One Workflow)

Right now you have two ways to get hello.py in front of VS Code. Pay attention to which one we’re using, because this is the workflow rule every assignment in this course depends on.

  • The wrong way — double-click hello.py in Finder/Explorer. VS Code opens with just that one file, disconnected from any folder. Python doesn’t know what directory to look in for data files. You won’t notice the problem today (there are no data files yet), but starting in Module 4 you’ll get FileNotFoundError messages that make no sense.
  • The right way — open the folder in VS Code first, then click the file from the sidebar. Python automatically uses that folder as its working directory. Your data files just work. This is how every professional Python developer works.

We’re going to do it the right way every single time, starting now. Even though hello.py has no data files and you could get away with the wrong way today, building the habit on day one means you never trip over the problem later.

To open the folder:

  1. In Finder/Explorer, navigate to is2053/start_here/
  2. Drag the start_here folder onto the VS Code icon — OR — open VS Code and use File → Open Folder, then pick start_here
  3. You should now see hello.py listed in the Explorer sidebar on the left. This means VS Code has the folder open as a workspace, and the file is accessible from inside it. That’s the workflow.
Step 2 — Write the program and run it locally

Now that the folder is open in VS Code, click hello.py in the Explorer sidebar on the left. It opens in the editor. Type exactly this line: 

print("Hello, World!")
match_case
Match the Cases and Punctuation Exactly

CodeGrade is going to compare what your program prints against what it expects, character by character. It is case-sensitive and punctuation-sensitive.

Type the line exactly as shown above:

  • Capital H in Hello, capital W in World
  • A comma after Hello, followed by a single space
  • An exclamation point at the end
  • Double quotes around the whole string

If you type print("hello world") or print("Hello world!") or even print("Hello,World!") (missing the space), the test will fail. Get used to this now: every assignment in this course is graded by a computer that does not interpret — only compare. Exact matters.

The easiest way to nail this: copy the spec, don’t retype it. When the instructions show you the exact text the program should print, select it, copy it, paste it into your code. Zero typos, zero case mistakes, zero missed punctuation. Professionals do this constantly — nobody manually retypes a string when they can copy it from the spec. Save your typing energy for the parts of the program that actually require thinking.

“Here’s a thing they don’t teach you in school but you learn fast in industry: the spec is the contract. When a client signs off on a build spec that says the output reads ‘Hello, World!’, that’s the deliverable. Not ‘hello, world’, not ‘Hello World’. Your job is to execute the spec, not relitigate it. The spec was decided above your pay grade.

If you think there’s a better way to do it, great — say so. Bring it up. Push back. Open the conversation. But until the spec gets formally changed, you build what the contract says. That’s the difference between a junior dev and a developer the client actually trusts with the keys.”

Danny Cavanaugh
Lead Developer, Bat City Collective

Save the file (Cmd + S on Mac, Ctrl + S on Windows). Then click the Play button in the upper-right corner of the editor. A terminal panel opens at the bottom of VS Code and you should see: 

Hello, World!
lightbulb
Your Output Is the Last Line — Ignore the Path Noise Above It

When you click the Play button, VS Code dumps a noisy first line into the terminal before your program’s actual output. The whole thing looks roughly like this:

PS C:\Users\you> & 'C:/Users/you/AppData/Local/Programs/Python/Python314/python.exe' 'c:/Users/you/Documents/is2053/start_here/hello.py'
Hello, World!

That first long line is just VS Code telling the terminal which Python it’s about to run and which file it’s feeding in — useful for troubleshooting later, but it is not your program’s output. Your output is the line that actually says Hello, World! — if you see that line, your program ran correctly. The path stuff above it is normal and will appear every time you click Play.

That’s your first Python program working locally. Now we send it to CodeGrade.

If the Play Button Doesn’t Work

Most Play button errors come down to one of two things: VS Code doesn’t know which Python to use, or Python isn’t on your PATH. Here’s how to figure out which one it is.

Step 1 — Check whether Python is on your PATH. Open a fresh terminal (Terminal from the Start menu on Windows, or Terminal.app on Mac — not the one inside VS Code) and type:

python --version

If you see something like Python 3.14.5, Python is installed and on your PATH. Skip to Step 3.

If you see “‘python’ is not recognized” (Windows) or “command not found” (Mac), Python is missing from your PATH. Continue to Step 2.

Step 2 — Get Python on your PATH (only if Step 1 failed).

Windows easy fix: Re-run the standalone installer from python.org (remember: skip the install manager button — see Checkpoint 3). If the installer shows the “Add Python to PATH” checkbox on the first screen, check it before clicking Install. This works for most students.

Didn’t see the PATH checkbox, or the reinstall didn’t fix it? Click to expand manual PATH instructions for Windows.

Some installer versions skip the PATH checkbox — especially if you previously had Python installed on this machine. Here’s how to add Python to your PATH yourself:

  1. Press the Windows key, type environment variables, and click Edit the system environment variables.
  2. In the dialog that opens, click Environment Variables… near the bottom.
  3. In the top section (User variables), find the variable named Path, select it, and click Edit.
  4. Click New and paste this (adjust 314 to match the version you installed — 313 for 3.13, etc.): 
    %LOCALAPPDATA%\Programs\Python\Python314\
  5. Click New again and add the Scripts folder: 
    %LOCALAPPDATA%\Programs\Python\Python314\Scripts\
  6. Click OK on every dialog to save.
  7. Close all open terminal windows and VS Code completely. PATH changes don’t apply to already-open windows.
  8. Reopen a terminal and type python --version again. You should see a version number now.

What’s PATH? When you type python in a terminal, your computer searches a list of folders for a program named python.exe. That list is called PATH. Adding Python’s install folder to PATH tells your computer where to find Python.

Installed Python somewhere else? Open File Explorer and search for python.exe. The folder where it lives is what you want to add to PATH.

Mac easy fix: The Mac installer handles PATH automatically, so a “command not found” error usually means the install didn’t complete. Re-download the .pkg from python.org and run it again. Make sure it finishes without errors.

Step 3 — Tell VS Code which Python to use. If python --version worked in Step 1 but the Play button still doesn’t, VS Code hasn’t picked an interpreter yet. Open the Command Palette with Cmd + Shift + P (Mac) or Ctrl + Shift + P (Windows), type Python: Select Interpreter, and pick the one with the highest version number (should be 3.14 or newer).

After any fix, close VS Code completely, reopen it, open the start_here folder again, and try the Play button. Still stuck? Drop a note in Discord #general or book a 1-on-1.
