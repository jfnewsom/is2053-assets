Section 1: Classes and Objects

Classes are blueprints. Objects are instances built from those blueprints. This is the core of OOP.

OOP Concepts
  • Object-Oriented Programming (OOP) — organizing code around objects that combine data and behavior
  • Object — a software entity that contains data (attributes) and methods
  • Class — a blueprint or template for creating objects
  • Instance — a specific object created from a class
  • Encapsulation — bundling data and methods together; hiding internal details
Class vs Object
  • Class — the definition/blueprint (like a recipe)
  • Object — a specific instance (like a cake made from the recipe)
  • Multiple objects — one class can create many objects
  • Each object is independent — changing one doesn’t affect others

Section 2: The __init__ Method and self

__init__ runs automatically when you create an object. self refers to that specific object.

The __init__ Method
  • __init__ — special method (constructor) that initializes a new object
  • Automatic call — Python calls __init__ when you create an object
  • Purpose — set up initial state (attribute values) of the object
  • Dunder method — double underscore on each side
The self Parameter
  • self — refers to the object being created or acted upon
  • First parameter — self is ALWAYS the first parameter in methods
  • Automatic passing — Python passes self; you don’t include it when calling
  • Setting attributes — self.name = name creates instance attribute
class City:
    def __init__(self, name, population):
        self.name = name
        self.population = population

Section 3: Accessors, Mutators, and Data Hiding

Getters return attribute values. Setters change them. This controls access to object data.

Accessor Methods (Getters)
  • Accessor — method that returns the value of an attribute
  • Naming convention — get_attribute_name() (e.g., get_name())
  • Purpose — provides controlled access to attribute values
Mutator Methods (Setters)
  • Mutator — method that changes the value of an attribute
  • Naming convention — set_attribute_name() (e.g., set_name())
  • Validation — setters can check values before accepting them
Data Hiding
  • Data hiding — preventing direct access to an object’s attributes
  • Single underscore (_) — “private by convention” (can still access)
  • Double underscore (__) — name mangling (harder to access)
  • Why hide data? — prevents invalid values, allows internal changes

Section 4: Inheritance

Inheritance lets a class receive attributes and methods from another class. Reuse code, don’t repeat it.

Inheritance Concepts
  • Inheritance — creating a new class based on an existing class
  • Superclass (parent class) — the class being inherited from
  • Subclass (child class) — the class that inherits
  • “is-a” relationship — subclass is a specialized type of superclass
  • Syntax — class CapitalCity(City):
What Gets Inherited
  • Attributes — subclass gets all superclass attributes
  • Methods — subclass gets all superclass methods
  • Adding new — subclass can add its own attributes and methods
  • Overriding — subclass can replace inherited methods
Calling super().__init__()
  • super() — returns reference to superclass
  • super().__init__() — calls superclass’s __init__ method
  • Why call it? — to initialize inherited attributes
class CapitalCity(City):
    def __init__(self, name, population, state):
        super().__init__(name, population)
        self.state = state

Section 5: Polymorphism

Polymorphism means “many forms.” Same method name, different behavior depending on the object.

Polymorphism Concepts
  • Polymorphism — ability of different objects to respond to same method call
  • Method overriding — subclass provides different implementation of inherited method
  • Override — subclass defines method with same name as superclass
  • Subclass version wins — when called on subclass object, subclass method runs
The isinstance() Function
  • isinstance(obj, Class) — returns True if obj is instance of Class
  • Works with inheritance — returns True for superclasses too

Section 7: Professional Practices

Good OOP design is the difference between maintainable code and spaghetti.

OOP Design Principles
  • Single Responsibility — each class should have one job
  • Favor composition over inheritance — use has-a before is-a
  • Use meaningful names — class names are nouns, methods are verbs
  • Hide implementation — expose what users need, hide how it works
Common Module 5 Mistakes
  • Forgetting self — all methods need self as first parameter
  • Missing __init__ call — subclass must call super().__init__() if needed
  • Confusing class and instance — City is class, austin is instance
  • Not using self. for attributes — name vs self.name
  • Wrong parentheses — City() creates object, City is the class
