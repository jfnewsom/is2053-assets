Step 1 — Open Folder, Not File (Your Day-One Workflow)

Right now you have two ways to get hello.py in front of VS Code. Pay attention to which one we’re using, because this is the workflow rule every assignment in this course depends on.

  • The wrong way — double-click hello.py in Finder/Explorer. VS Code opens with just that one file, disconnected from any folder. Python doesn’t know what directory to look in for data files. You won’t notice the problem today (there are no data files yet), but starting in Module 4 you’ll get FileNotFoundError messages that make no sense.
  • The right way — open the folder in VS Code first, then click the file from the sidebar. Python automatically uses that folder as its working directory. Your data files just work. This is how every professional Python developer works.

We’re going to do it the right way every single time, starting now. Even though hello.py has no data files and you could get away with the wrong way today, building the habit on day one means you never trip over the problem later.

To open the folder:

  1. In Finder/Explorer, navigate to is2053/start_here/
  2. Drag the start_here folder onto the VS Code icon — OR — open VS Code and use File → Open Folder, then pick start_here
  3. You should now see hello.py listed in the Explorer sidebar on the left. This means VS Code has the folder open as a workspace, and the file is accessible from inside it. That’s the workflow.
Step 2 — Write the program and run it locally

Now that the folder is open in VS Code, click hello.py in the Explorer sidebar on the left. It opens in the editor. Type exactly this line: 

print("Hello, World!")
match_case
Match the Cases and Punctuation Exactly

CodeGrade is going to compare what your program prints against what it expects, character by character. It is case-sensitive and punctuation-sensitive.

Type the line exactly as shown above:

  • Capital H in Hello, capital W in World
  • A comma after Hello, followed by a single space
  • An exclamation point at the end
  • Double quotes around the whole string

If you type print("hello world") or print("Hello world!") or even print("Hello,World!") (missing the space), the test will fail. Get used to this now: every assignment in this course is graded by a computer that does not interpret — only compare. Exact matters.

The easiest way to nail this: copy the spec, don’t retype it. When the instructions show you the exact text the program should print, select it, copy it, paste it into your code. Zero typos, zero case mistakes, zero missed punctuation. Professionals do this constantly — nobody manually retypes a string when they can copy it from the spec. Save your typing energy for the parts of the program that actually require thinking.

“Here’s a thing they don’t teach you in school but you learn fast in industry: the spec is the contract. When a client signs off on a build spec that says the output reads ‘Hello, World!’, that’s the deliverable. Not ‘hello, world’, not ‘Hello World’. Your job is to execute the spec, not relitigate it. The spec was decided above your pay grade.

If you think there’s a better way to do it, great — say so. Bring it up. Push back. Open the conversation. But until the spec gets formally changed, you build what the contract says. That’s the difference between a junior dev and a developer the client actually trusts with the keys.”

Save the file (Cmd + S on Mac, Ctrl + S on Windows). Then click the Play button in the upper-right corner of the editor. A terminal panel opens at the bottom of VS Code and you should see: 

Hello, World!

Your Output Is the Last Line — Ignore the Path Noise Above It

When you click the Play button, VS Code dumps a noisy first line into the terminal before your program’s actual output. The whole thing looks roughly like this:

PS C:\Users\you> & 'C:/Users/you/AppData/Local/Programs/Python/Python314/python.exe' 'c:/Users/you/Documents/is2053/start_here/hello.py'
Hello, World!

That first long line is just VS Code telling the terminal which Python it’s about to run and which file it’s feeding in — useful for troubleshooting later, but it is not your program’s output. Your output is the line that actually says Hello, World! — if you see that line, your program ran correctly. The path stuff above it is normal and will appear every time you click Play.

That’s your first Python program working locally. Now we send it to CodeGrade.

If the Play Button Doesn’t Work

Most Play button errors come down to one of two things: VS Code doesn’t know which Python to use, or Python isn’t on your PATH. Here’s how to figure out which one it is.

Step 1 — Check whether Python is on your PATH. Open a fresh terminal (Terminal from the Start menu on Windows, or Terminal.app on Mac — not the one inside VS Code) and type:

python --version

If you see something like Python 3.14.5, Python is installed and on your PATH. Skip to Step 3.

If you see “‘python’ is not recognized” (Windows) or “command not found” (Mac), Python is missing from your PATH. Continue to Step 2.

Step 2 — Get Python on your PATH (only if Step 1 failed).

Windows easy fix: Re-run the standalone installer from python.org (remember: skip the install manager button — see Checkpoint 3). If the installer shows the “Add Python to PATH” checkbox on the first screen, check it before clicking Install. This works for most students.

Didn’t see the PATH checkbox, or the reinstall didn’t fix it? Click to expand manual PATH instructions for Windows.

Some installer versions skip the PATH checkbox — especially if you previously had Python installed on this machine. Here’s how to add Python to your PATH yourself:

  1. Press the Windows key, type environment variables, and click Edit the system environment variables.
  2. In the dialog that opens, click Environment Variables… near the bottom.
  3. In the top section (User variables), find the variable named Path, select it, and click Edit.
  4. Click New and paste this (adjust 314 to match the version you installed — 313 for 3.13, etc.): 
    %LOCALAPPDATA%\Programs\Python\Python314\
  5. Click New again and add the Scripts folder: 
    %LOCALAPPDATA%\Programs\Python\Python314\Scripts\
  6. Click OK on every dialog to save.
  7. Close all open terminal windows and VS Code completely. PATH changes don’t apply to already-open windows.
  8. Reopen a terminal and type python --version again. You should see a version number now.

What’s PATH? When you type python in a terminal, your computer searches a list of folders for a program named python.exe. That list is called PATH. Adding Python’s install folder to PATH tells your computer where to find Python.

Installed Python somewhere else? Open File Explorer and search for python.exe. The folder where it lives is what you want to add to PATH.

Mac easy fix: The Mac installer handles PATH automatically, so a “command not found” error usually means the install didn’t complete. Re-download the .pkg from python.org and run it again. Make sure it finishes without errors.

Step 3 — Tell VS Code which Python to use. If python --version worked in Step 1 but the Play button still doesn’t, VS Code hasn’t picked an interpreter yet. Open the Command Palette with Cmd + Shift + P (Mac) or Ctrl + Shift + P (Windows), type Python: Select Interpreter, and pick the one with the highest version number (should be 3.14 or newer).

After any fix, close VS Code completely, reopen it, open the start_here folder again, and try the Play button. Still stuck? Drop a note in Discord #general or book a 1-on-1.
