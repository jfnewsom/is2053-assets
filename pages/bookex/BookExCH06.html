Files and Exceptions
Chapter 6  —  10 Programs

Work through each Gaddis program in order. Chapter 6 is where your programs start talking to the file system. Type every program by hand and actually run each one — you need to see the files get created and read back. Programs 1 through 7 build the foundation; Programs 8 through 10 introduce the with statement, append mode, and exception handling.

Learning Objectives
  • Open a file in write mode and write strings with newline characters
  • Open a file in read mode and read its entire contents
  • Read a file one line at a time using readline()
  • Use rstrip() to remove newline characters from lines read from a file
  • Convert numbers to strings before writing, and strings back to numbers after reading
  • Use the with statement for automatic file closing
  • Open a file in append mode to add records without overwriting existing data
  • Use try-except to handle FileNotFoundError gracefully
Key Concepts

Chapter 6 covers file modes ('r', 'w', 'a'), write() and readline(), the end-of-file sentinel pattern, rstrip(), type conversion for file I/O, the with statement, and try-except for FileNotFoundError.

error
Mode 'w' Overwrites the Entire File

Opening a file in 'w' mode destroys all existing content and starts fresh. If philosophers.txt already existed, it is gone. This is intentional here, but it is a dangerous habit to get into for data you want to keep. You will use append mode ('a') in Checkpoint 8 to add without overwriting.

info
readline() Keeps the Newline Character

Each readline() call returns the line including its trailing \n. When you then print() that string, print() adds its own newline on top — resulting in a blank line between each name. Checkpoint 5 shows how to strip that extra \n before printing.

lightbulb
write() Does Not Add Newlines Automatically

Unlike print(), write() puts exactly what you give it and nothing more. If you write name1 without + '\n', all three names end up on a single line in the file. Always append '\n' when you want each item on its own line.

warning
write() Only Accepts Strings

Trying to write an integer directly — outfile.write(num1) — raises a TypeError. Files store text, not numbers. Use str(num1) to convert first. When reading the number back in Checkpoint 7, you then reverse this with int().

info
int() and readline() Can Be Combined

int(infile.readline()) reads a line and converts it to an integer in one step. This works because int() strips the trailing \n automatically when it parses the string. You could also do readline() then rstrip() then int() — same result, just more steps.

lightbulb
with Closes the File Automatically

When the with block ends (when the code indentation returns to the main() level), Python closes the file automatically — even if an error occurs inside the block. This is the preferred way to handle files in Python and eliminates the need for a separate close() call.

show_coffee_records.py
Program 6-19 (6th Ed) • Program 6-16 (5th Ed)
What You'Re Building

A program that reads coffee.txt and displays every record using the read-process-read loop pattern: read the first description before the loop, then loop while the description is not an empty string, process the record, and read the next description at the bottom of the loop. An empty string from readline() signals the end of the file.

Step-By-Step Instructions
  1. Open your textbook to Program 6-19 (6th Ed) or Program 6-16 (5th Ed).
  2. Create a new file named show_coffee_records.py.
  3. Type the program as shown, paying careful attention to where readline() is called.
  4. Make sure add_coffee_record.py has been run first to create coffee.txt.
  5. Run it and confirm all your coffee records appear.
warning
The Read-Process-Read Pattern Is Easy to Get Wrong

The loop reads descr before the while, processes inside the loop, then reads the next descr at the bottom. If you move that last readline() outside the loop or forget it, you get an infinite loop or read the wrong data. The structure is: read → while not empty → process → read next.

What You'Re Building

A file viewer that asks the user for a filename, then uses try-except to attempt opening and displaying it. If the file doesn't exist, the except FileNotFoundError block catches the error and prints a friendly message instead of crashing. This is the standard pattern for any program that opens files by name.

Step-By-Step Instructions
  1. Open your textbook to Program 6-28 (6th Ed) or Program 6-25 (5th Ed).
  2. Create a new file named display_file2.py.
  3. Type the program as shown.
  4. Run it and enter philosophers.txt — confirm the contents appear.
  5. Run it again and enter a filename that does not exist — confirm the error message appears instead of a crash.
lightbulb
try-except Is How You Handle Predictable Errors

Code in the try block runs normally. If a FileNotFoundError occurs, Python jumps to the except block instead of crashing. The program continues cleanly after the except block. This pattern works for any predictable error — not just file errors.
