Inheritance
Chapter 11  —  4 Programs

Chapter 11 extends what you built in Chapter 10. If classes are blueprints, inheritance lets you build a new blueprint by extending an existing one. A Car is an Automobile with extra attributes. A Dog is a Mammal that makes a specific sound. Programs 11-1 and 11-2 build a vehicle hierarchy and are merged into one file. Program 11-3 demonstrates how to use those classes. Programs 11-9 and 11-10 demonstrate method overriding: the ability of a subclass to replace a superclass method with its own version. This is what makes polymorphism work.

Learning Objectives
  • Define a superclass with private attributes, a constructor, and getter/setter methods
  • Define subclasses using class SubClass(SuperClass): syntax
  • Call the superclass constructor from a subclass using SuperClass.__init__(self, args)
  • Override a superclass method in a subclass
  • Create objects from both superclasses and subclasses and call their methods
Key Concepts

Chapter 11 covers inheritance (Section 11.1), calling the superclass constructor (Section 11.1), and method overriding (Section 11.2). Programs 11-1 through 11-3 demonstrate a vehicle hierarchy. Programs 11-9 and 11-10 demonstrate overriding with animals.

Programs 11-1 and 11-2: vehicles.py

Programs 11-1 and 11-2 merged — Vehicle Hierarchy
These two Gaddis programs are combined into one file. Program 11-1 defines the Automobile superclass. Program 11-2 defines Car, Truck, and SUV as subclasses. This file contains class definitions only, with no main().

What You're Building
  1. Define the Automobile superclass with an __init__ that accepts make, model, mileage, and price. Store all four in private attributes (self.__make, etc.). Write getter and setter methods for all four attributes.
  2. Define the Car subclass: class Car(Automobile):. Its __init__ accepts all Automobile parameters plus doors. Call the superclass constructor: Automobile.__init__(self, make, model, mileage, price). Store self.__doors = doors. Add getter and setter for doors.
  3. Define the Truck subclass the same way, but with a drive_type attribute (e.g., 'Two Wheel Drive') instead of doors.
  4. Define the SUV subclass with a pass_cap (passenger capacity) attribute.
Why This Matters

The subclass constructor calls Automobile.__init__(self, make, model, mileage, price) to initialize the inherited attributes. Without this call, the private attributes defined in Automobile.__init__ would never be created. The subclass would have no __make, __model, etc. Every subclass constructor must call the superclass constructor when the superclass uses __init__ to set up private attributes.

Tips and Pitfalls
MistakeWhat Goes WrongFix
Forgetting to call Automobile.__init__() in Car.__init__()Inherited attributes never created. AttributeError when getters are calledAlways call superclass __init__ first thing in the subclass constructor
Calling super().__init__() instead of Automobile.__init__(self, ...)Works in Python 3 but Gaddis uses the explicit form. Match the textbookUse Automobile.__init__(self, make, model, mileage, price)
Redefining inherited getter/setter methods in the subclassRedundant and creates confusion about which version runsOnly define what is new or overridden in the subclass

Program 11-3: car_demo.py

Program 11-3 — Using the Vehicle Classes
This program imports vehicles and creates a Car object, demonstrating that inherited methods and subclass-specific methods both work through the same object.

What You're Building
  1. Import vehicles.
  2. Write main(). Get make, model, mileage, price, and number of doors from the user.
  3. Create a Car object: car = vehicles.Car(make, model, mileage, price, doors).
  4. Display the car's data by calling all five getter methods: the four inherited from Automobile and get_doors() from Car.
  5. Add the if __name__ == '__main__': guard and call main().
Why This Matters

The Car object has access to both get_make() (inherited from Automobile) and get_doors() (defined in Car). You call both using the same dot notation — the inheritance is invisible to the caller. This is the design goal: a Car object behaves like an Automobile and can also do car-specific things.

Program 11-9: animals.py

Program 11-9 — Method Overriding
Define a Mammal superclass and two subclasses (Dog and Cat) that override the make_sound() method. This file contains class definitions only.

What You're Building
  1. Define class Mammal:. Its __init__ accepts a species parameter and stores it in self.__species. Write show_species(self) that prints the species. Write make_sound(self) that prints 'Grrrrr'.
  2. Define class Dog(Mammal):. Its __init__ takes no parameters other than self. Call Mammal.__init__(self, 'Dog'). Override make_sound(self) to print 'Woof! Woof!'.
  3. Define class Cat(Mammal): the same way, with 'Cat' as the species and 'Meow' as the sound.
  4. This file has no main() and no if __name__ guard. It is a class definition file.
Why This Matters

Method overriding is the core of polymorphism. Both Dog and Cat inherit show_species() from Mammal and use it without change. But each provides its own make_sound(). When you call dog.make_sound(), Python looks at the Dog class first and finds the override. The superclass version is never reached. This means you can write one loop that calls make_sound() on a list of mixed animals and get the right sound for each one automatically.

Program 11-10: polymorphism_demo.py

Program 11-10 — Polymorphism in Action
This program imports animals and creates a list of animal objects. It then loops through the list calling make_sound() on each one, demonstrating that the same method call produces different behavior depending on the object type.

What You're Building
  1. Import animals.
  2. Write main(). Create a list containing one Mammal object, one Dog object, and one Cat object.
  3. Write a for loop that iterates through the list. For each object, call show_species() and then make_sound().
  4. Add the if __name__ == '__main__': guard and call main().
  5. Run it and observe: the same make_sound() call produces three different outputs depending on the object.
Why This Matters

This is polymorphism: one interface, multiple behaviors. The loop does not know or care what type each animal is. It calls make_sound() and gets the right result. This is the same principle behind your game's City class in Module 5. When you call a method on a city object, the object itself knows what to do. You do not need a separate if/elif chain for each city type.
