Lists to the Rescue
MODULE 3 • WEEK 2 • ENHANCE • CHAPTERS 7: LISTS AND TUPLES
Overview

Your save file works, but writing six separate write() calls for six variables already feels tedious. Now imagine needing to save twenty cities. Or fifty hazards. That approach doesn't scale. This week you discover the tool that makes all of that elegant: lists. You will replace your if/elif destination chains with parallel lists and index lookups, add two tracking lists that grow automatically as the player travels, and upgrade your save/load system to persist those lists. You will also write three new utility functions into function_library.py — the first time the library grows this module.

"Every if/elif chain you write is a promise to maintain it forever. Lists let you make that promise once and never revisit it. That is what scaling looks like."
Priya Venkatesh
Lead Architect, Bat City Collective

What parallel lists look like

Position 0 in each list belongs together, position 1 belongs together, and so on. The index is the connection between them:

  • SA_DESTINATIONS[0] is 'Corpus Christi', SA_DISTANCES[0] is 150, SA_NOTES[0] is 'BEACH!'
  • SA_DESTINATIONS[1] is 'Houston', SA_DISTANCES[1] is 200, SA_NOTES[1] is 'Gateway East'
  • SA_DESTINATIONS[2] is 'Austin', SA_DISTANCES[2] is 80, SA_NOTES[2] is 'Keep it Weird'
  • SA_DESTINATIONS[3] is 'Laredo', SA_DISTANCES[3] is 160, SA_NOTES[3] is 'Dead End'

Checkpoint
2
Replace if/elif Destination Selection with List Indexing
What You're Building

This is the magic moment. The if/elif chain that selected a destination based on the player's menu choice gets replaced with a single index lookup. The player's choice (1–4) maps directly to an index (0–3) via choice - 1.

Step-by-Step Instructions
  1. Find the section of your game loop where the player's menu choice determined which city and distance to use. It probably looks like a chain of if choice == 1: ... elif choice == 2: ... blocks.
  2. Replace that entire chain with index lookups. Compute the index once: index = choice - 1, then use it to pull both values: destination = SA_DESTINATIONS[index] and distance = SA_DISTANCES[index].
  3. If your code displays a city note or tagline, pull that from SA_NOTES[index] as well.
  4. Test by running the game and selecting each of the four destination choices in turn. The destination name, distance, and note should all be correct.
"You just deleted fifteen lines of if/elif and replaced them with two. That is not a trick — that is how professional code actually works. One lookup, any size list."
Terrence "T-Bone" Booker
Senior Developer, Bat City Collective
Off-by-one: choice vs index

Player choices start at 1 (because menus do), but list indices start at 0 (because Python does). Always subtract 1: index = choice - 1. If you forget, choice 1 gives you the second city, and choice 4 crashes with an IndexError.

What You Should Have
  1. The destination if/elif chain is completely removed
  2. Destination and distance are set via SA_DESTINATIONS[index] and SA_DISTANCES[index]
  3. All four destination choices work correctly in a test run

Add the visited_cities and hazards_list Tracking Lists
What You're Building

Now you add two empty lists to the game state — one that grows every time the player arrives in a city, and one that grows every time a hazard is encountered. By the time the final trip report prints, both lists contain a complete record of the journey.

Step-by-Step Instructions
  1. In the game state initialization section (after the main menu, once you know whether this is a new or loaded game), add two list variables. For a new game, initialize them as: visited_cities = ['San Antonio'] and hazards_list = []. San Antonio is pre-loaded because the player starts there.
  2. Inside the game loop, after the player arrives at their destination, append that destination to the list: visited_cities.append(destination).
  3. Before appending, check whether the player has already visited that city using the in operator. If they have, print a message like 'You have been to Houston before!' — then still append, since the list tracks the route, not unique cities.
  4. Inside the hazard handling block, when a hazard is triggered, append the hazard name string to hazards_list: hazards_list.append(hazard).
  5. In the final trip report, replace any hardcoded stat output with list-based display. Loop through visited_cities using range(len(visited_cities)) to print a numbered route. Use len(hazards_list) to report the hazard count.

Upgrade save_game() and load_game() for the Enhanced Format
What You're Building

Your save file currently holds six lines of scalar data. This checkpoint upgrades it to also persist the two tracking lists. Cities go after the six stats, followed by a ---HAZARDS--- separator line, followed by hazards. The load function reads cities until it hits the separator, then reads hazards until the file ends.

Step-by-Step Instructions
  1. Update save_game() to accept two additional parameters: visited_cities and hazards_list. Write the six scalar variables exactly as before.
  2. After the six scalars, loop through visited_cities and write each city on its own line.
  3. Write the separator line: outfile.write('---HAZARDS---\n')
  4. Loop through hazards_list and write each hazard on its own line. Then close the file.
  5. Update load_game() to read back the two lists. After reading the six scalars, read cities into a local list using a while loop — keep reading until you hit the separator line '---HAZARDS---' or an empty string.
  6. After the separator, read hazards into a second local list using another while loop until you reach an empty string (end of file).
  7. Return a tuple that now includes all eight values: the six scalars plus the two lists.
  8. Update the main menu's Load Game branch to unpack all eight values from the tuple.

  • Program 6-19 — show_coffee_records.py (Step 2) — Shows the while line != '' pattern for reading variable-length data from a file.

  4. Add file_exists(filename) — attempts to open the file in read mode inside a try block and returns True. Returns False in the except FileNotFoundError block.
