What You're Building

A clean refactoring of two functions out of lab-3-1.py and into function_library.py, along with the constant they depend on. You'll touch four things: the function definitions, the constant declaration, the call sites, and any leftover references.

Why This Matters

Moving a function is more than copy-paste. Every function has dependencies — constants it reads, modules it imports, functions it calls. When the function moves, those dependencies have to travel with it OR be passed in as parameters. save_game() depends on SAVE_FILE; they move together. Then every call site (in lab-3-2.py) has to be updated to use the new path: function_library.save_game(...) instead of just save_game(...). Miss one call site and you get NameError at runtime. This is the refactoring practice you'll do hundreds of times in your career.

Step-by-Step Instructions
  1. Open both files: lab-3-1.py (which you copied as lab-3-2.py in CP1) and function_library.py side by side
  2. In lab-3-2.py, locate the SAVE_FILE = 'savegame.txt' line, the entire def save_game(...): function block (including its docstring), and the entire def load_game(): function block
  3. In function_library.py, find a good spot for them — right after the existing validation functions is fine. Add a header comment: # SAVE/LOAD FUNCTIONS (Promoted from lab-3-1.py in Lab 3-2)
  4. Move SAVE_FILE to function_library.py (place it near the top of the file with the other constants), then delete it from lab-3-2.py
  5. Move the save_game() function: cut the entire definition (including docstring) from lab-3-2.py and paste it into function_library.py under your new header comment
  6. Move the load_game() function the same way
  7. Update the call sites in lab-3-2.py: search for save_game( and prefix each call with function_library. so it becomes function_library.save_game(. Do the same for load_game()
  8. Check for leftover references: search lab-3-2.py for any remaining SAVE_FILE — that constant lives only in the library now
  9. Run lab-3-2.py. Play through one game, save mid-trip in Houston, quit, restart, load. The behavior should be identical to Lab 3.1 from the user's perspective. The difference is invisible from outside — that's how you know the refactor succeeded

Why This Matters

Parallel lists are the simplest way to associate related pieces of data. Same index across all three lists = same destination. SA_DESTINATIONS[0] is 'Corpus Christi', SA_DISTANCES[0] is 150, SA_NOTES[0] is 'BEACH!' — together they describe the first destination. Adding a new destination becomes a one-line change to each list. Compare that to the current pattern where you'd add an elif branch, an if branch in the redirect logic, and a menu print statement. Parallel lists scale; if/elif chains don't.

Menu still shows old format without the notes column at one or more cities Check CP4 and CP6 — you may have refactored SA but not the Houston or Austin menu. After CP6, all three cities should go through function_library.display_destinations(...).
TAILS gate always sends the player to the same destination Check CP5 — get_redirect needs random.randint AND needs .copy() on both lists. Without copy, you mutate the originals and eventually only one option remains.
visited_cities empty at end of game Check CP7 — you forgot the .append(current_city) after the travel loop ends. The list is initialized but nothing's adding to it.
too many values to unpack error after loading a saved game Check CP8 — you extended load_game to return 11 values, but the unpack site in main() still expects 9. Update the unpack to include visited_cities and hazards_list.
Save file shows the ---LISTS_END--- marker but loading doesn't find it Check CP8 — your while condition in load_game probably just checks line != ''. It also needs line != '---LISTS_END---'.
Trip Summary lists are missing the divider line or trailing space Check CP9 — display_numbered_list needs to print: title, then '-' * len(title), then the numbered items, then a blank print() at the end.
Lists persist across games (e.g., game 2 shows game 1's hazards too) Check CP7 — the list initializations need to be INSIDE the play-again loop's per-game reset block, not before the loop.
Loaded game crashes with ValueError or wrong field counts Check CP8 — if your save file is from before you finished CP8 (or from Lab 3.1 directly), it has the wrong number of lines for the new format. Delete savegame.txt and start fresh. The library's load_game() handles missing files cleanly, but a partial or old-format file may have parse errors. If a fresh save then loads cleanly, the fix is good. If it still crashes, your load_game() isn't catching ValueError — update the except clause to except (FileNotFoundError, ValueError):.
