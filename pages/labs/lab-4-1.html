Module 4 — Week 1 — BUILD Week

Chapters: 8–9 — More About Strings, Dictionaries and Sets

Your Module 3 code is functional, but it has a problem that every working developer eventually hits: hardcoded data. You have 20+ individual distance constants, city-type checks baked into if/elif chains, and no easy way to add a new city without touching five different places in the file. This week, you fix all of that in one move. CITY_DISTANCES and CITY_INFO are two dictionaries that replace every one of those constants. Adding a new city goes from a multi-file, multi-section change to a single dictionary entry. That's what data-driven design looks like in practice.

Checkpoint
1
Build CITY_DISTANCES
What You're Building

You are creating a nested dictionary that maps every city to its available destinations and the driving distance in miles. Once this is in place, CITY_DISTANCES['San Antonio']['Austin'] returns 80 — no constant required.

Textbook Reference
  • Program 9-1 — card_dealer.py — Shows dictionary creation, key access with bracket notation, and looping through keys. The same patterns apply directly to CITY_DISTANCES.
Step-by-Step Instructions
  1. Delete (or comment out) all of your individual distance constants from Module 3 — the lines like SAT_AUS = 80, SAT_HOU = 200, and so on.
  2. Near the top of your file, create CITY_DISTANCES as an empty dictionary: CITY_DISTANCES = {}.
  3. Add San Antonio as the first outer key. Its value is another dictionary with each destination city as a key and the mileage as an integer value.
  4. Add the remaining nine cities as outer keys, each with their own inner destination dictionary. Use the city and mileage data provided in the code plan.
  5. Test a few lookups in isolation before continuing: print(CITY_DISTANCES['San Antonio']['Austin']) should print 80.
Build It in Pieces

Don't try to write all ten cities at once. Add San Antonio, test it, then add Houston, test again. A working two-city dictionary is better than a broken ten-city one.

Keys Are Case-Sensitive

Dictionary keys must match exactly — capitalization, spaces, and all. 'San Antonio' and 'san antonio' are different keys. Pick one convention and stick to it throughout the entire file.

What You're Building

You are creating a second dictionary that stores each city's type (WIN, HUB, GATEWAY, etc.) and its tagline. This replaces every hardcoded city-type check in your game loop with a clean dictionary lookup.

Step-by-Step Instructions
  1. Below your CITY_DISTANCES constant, create CITY_INFO = {}.
  2. For each of the ten cities, add an entry where the key is the city name (spelled exactly as in CITY_DISTANCES) and the value is a dictionary with two keys: 'type' and 'tagline'.
  3. Use these city types and taglines from the code plan: San Antonio (START, 'Home of the Alamo'), Houston (GATEWAY, 'Space City'), Austin (CORRIDOR, 'Keep it weird'), Corpus Christi (WIN, 'The Sparkling City by the Sea'), Galveston (WIN, 'The Playground of the South'), Laredo (DEAD_END, 'Gateway to Mexico'), El Paso (WRONG_WAY, 'Closer to California than Houston'), Dallas (HUB, 'Where everything is bigger'), Fort Worth (HUB, 'Where the West begins'), Texarkana (DEAD_END, 'Two states at once').
  4. Test a lookup: print(CITY_INFO['Austin']['tagline']) should print Keep it weird.
Why a Nested Dict Instead of Two Separate Ones?

You could have separate CITY_TYPES and CITY_TAGLINES dictionaries, but grouping all city data under one outer key keeps related information together. When you add a new city, you add one entry — not two.

  • Easier to maintain — one place to update
  • Easier to extend — add a new field without touching existing code
  • Clearer intent — CITY_INFO['Austin'] returns everything about Austin
What You Should Have

After this checkpoint, you have two dictionaries defined near the top of your file. Neither is connected to the game loop yet — that happens in CP3. Run the file to confirm it loads without errors, then move on.

Checkpoint
3
Refactor the Helper Functions
What You're Building

Now you wire the dictionaries into the game by rewriting four helper functions. Three of them (display_destinations, is_win_city, is_dead_end) already exist in your code from Module 3. You are replacing their implementations. One (get_destination_by_number) is new.

Step-by-Step Instructions
  1. display_destinations(current_city): Use CITY_DISTANCES[current_city] to get the destination dictionary. Loop through it with a counter to display each option as a numbered menu. Return list(CITY_DISTANCES[current_city].keys()) so the game loop knows what options are available.
  2. get_destination_by_number(current_city, choice_num): Get list(CITY_DISTANCES[current_city].keys()), then index into it with choice_num - 1. Return the destination city name and its mileage as two separate values.
  3. is_win_city(city_name): Use CITY_INFO.get(city_name, {}) to safely fetch the city's data, then check if city_data.get('type', '') equals 'WIN'. Return True or False.
  4. is_dead_end(city_name): Same pattern as is_win_city, but check if the type is in ['DEAD_END', 'WRONG_WAY'].
  5. Update your game loop to display the city tagline from CITY_INFO at the start of each turn: tagline = CITY_INFO.get(current_city, {}).get('tagline', '').
  6. Run the game and play through at least one full turn. Confirm the destination menu appears correctly and distances match the dictionary values.
Use .get() for Safe Lookups

When looking up a city in CITY_INFO, use .get(city_name, {}) instead of direct bracket access. If a city somehow isn't in the dictionary, you get an empty dict back instead of a KeyError that crashes the program.

  • CITY_INFO['Atlantis'] — raises KeyError, program crashes
  • CITY_INFO.get('Atlantis', {}) — returns empty dict, program continues

Tips & Pitfalls
Mistake What Goes Wrong Fix
Using the old constants alongside the new dict Confusing source of truth — distance from where? Delete or comment out all old constants before testing
CITY_DISTANCES[current_city] raises KeyError City name in game state doesn't match dict key exactly Print current_city and compare to dict keys character by character
get_destination_by_number returns wrong city Off-by-one error — player enters 1, index is 0 Use choice_num - 1 when indexing into the keys list
display_destinations shows no options Returned list is empty or function isn't being called Check that current_city is spelled exactly as a key in CITY_DISTANCES
