Key Concepts

Two new tools arrive together. .strip().split(',') is the standard CSV-parsing pattern from Chapter 8: .strip() removes the newline at the end of each line, .split(',') breaks the comma-separated values into a list of fields. Dictionaries (Chapter 9) replace parallel lists for any data shape where the keys are meaningful. A nested dictionary is just a dictionary whose values are themselves dictionaries: distances['San Antonio'] is itself a dictionary mapping destinations to miles. The in operator works on dictionaries too. 'Houston' in distances checks if Houston is a key. This is how the loader builds the nested structure: it checks if city1 not in distances: before creating the inner dictionary.

Load City Distances
What You're Building

A new function in function_library.py named load_city_distances(filename). It opens the named text file, reads it one line at a time, parses each line into a city pair plus a distance, and builds a nested dictionary where the outer key is the starting city, the inner key is the destination, and the inner value is the distance in miles.

Why This Matters

Lab 3.3 declared one constant per route at the top of lab-3-3.py: SAT_TO_CORPUS = 150, SAT_TO_HOUSTON = 200, and so on. That works for three cities and ten edges. It does not work for ten cities and twenty-four edges, which is what Module 4 hands you. The fix is to keep the data in a text file and write one function that reads it. Adding a city later means adding a line to the text file, not editing code. Same data, different storage shape, dramatically less work downstream.

"If you had to add twenty more cities to this game next week, how long would it take? If the answer is 'just add twenty lines to a text file', the design is right. If the answer is 'edit the code in five places', the design is wrong. Pick the data shape that makes the next change small."
Priya Venkatesh
Project Manager, Bat City Collective
How the road network is modeled

The 24 routes in city_distances.txt are the cities and distances you can drive on a Texas interstate. San Antonio reaches Houston (I-10), Austin (I-35), Laredo (I-35), El Paso (I-10), and Corpus Christi (I-37). Houston reaches Galveston (I-45) and Dallas (I-45). Austin reaches Dallas and Fort Worth via the I-35E and I-35W split. Dallas and Fort Worth connect via I-30; Dallas reaches Texarkana via I-30; Fort Worth reaches El Paso via I-20 and I-10. There is no interstate between Austin and Houston, between Laredo and El Paso, or between Corpus Christi and Laredo, so those routes do not appear in the data. To get between two cities that aren't connected directly, the player has to drive through an intermediate city — usually San Antonio.

How the road network is modeled

The city graph in city_distances.txt only includes pairs of cities connected by a single continuous interstate corridor. San Antonio to Houston is on the menu because I-10 runs between them. Austin to Houston is not on the menu, even though the cities are 165 miles apart, because the natural route is US-290 — a state highway, not an interstate. Laredo and Corpus Christi only connect back to San Antonio for the same reason: no direct interstate between them and the rest of the network. If a route feels missing, that's the constraint at work, not a bug in the data file.

Step-by-Step Instructions
  1. Open function_library.py. At the bottom, add a new section header comment block titled CITY DATA LOADERS.
  2. Define load_city_distances(filename) with a full docstring describing what it returns.
  3. Inside the function, create an empty dictionary called distances.
  4. Open the file with infile = open(filename, 'r').
  5. Loop through the file with for line in infile:. For each line, call parts = line.strip().split(',') to split it into a list of three strings.
  6. Assign city1 = parts[0], city2 = parts[1], and miles = int(parts[2]). Only the third part needs int() — the first two are city names.
  7. If city1 is not yet a key in distances, set distances[city1] = {}. Then set distances[city1][city2] = miles.
  8. After the loop, close the file with infile.close() and return distances.

Tips & Pitfalls
Forgetting .strip() Each line ends with a newline character. Without .strip(), your last 'destination' key has a hidden \n on the end, and every lookup mysteriously fails.
Casting parts[0] or parts[1] to int Only the distance is a number. City names are strings. Doing int(parts[0]) throws ValueError the moment you hit a non-numeric city name.
Skipping the if city1 not in distances: guard Without it, the very first time you see a city you also try to assign distances[city1][city2], which fails because distances[city1] does not exist yet.
Trying to use with open(...) as ... with statements are not in scope for this lab. Use the plain open()/close() pattern from Chapter 6.

Load City Info
What You're Building

A second loader function, load_city_info(filename), that parses city_info.txt into a dictionary of dictionaries: each city name maps to an inner dictionary with two keys, 'type' (one of START, HUB, GATEWAY, CORRIDOR, WRONG_WAY, DEAD_END, WIN) and 'tagline' (the city's short slogan).

Why This Matters

City info is the metadata that drives game logic. 'WIN' cities end the trip with a beach celebration. 'DEAD_END' and 'WRONG_WAY' cities prompt the player to return to San Antonio. 'GATEWAY', 'HUB', 'CORRIDOR' are all just intermediate cities for now. Bundling all of this into one dictionary keyed by city name means the lab file does not need branching logic per city; it just asks the info dictionary what to do.

"Notice that the loader pattern is identical to the one you wrote ten minutes ago. Same .strip().split(','). Same per-line dictionary assignment. Different data shape on the output side, same parsing pattern on the input side. That is what good design feels like. The boring part is reusable."

Step-by-Step Instructions
  1. Below load_city_distances in function_library.py, define load_city_info(filename) with a full docstring.

Keeping the function names stable across a data-shape refactor is a real-world discipline. The rest of the lab file calls display_destinations(...) the same way it always has. "When you refactor a function, keep the public name and shape stable if you can. Common failure patterns and where they came from:

Symptom Where to Look
KeyError: 'San Antonio' on the first menu display Check CP1 (Load City Distances) — the loader is not inserting 'San Antonio' as a top-level key. Verify the if city1 not in distances: guard.
FileNotFoundError: city_distances.txt Check CP4 (Lab Skeleton). The data files must be in the same folder as lab-4-1.py when you run it locally. CodeGrade provides them as fixtures.

ValueError: invalid literal for int() when loading distances Check CP1. After parts = line.strip().split(','), only parts[2] gets cast to int. parts[0] and parts[1] stay as strings.
