Overview

All semester, you've been building Deep in the Heart: A Lone Star Journey with dictionaries, functions, and files. It works — but the code is scattered. City data lives in one place, distance lookups in another, and game logic somewhere else. This week, everything changes. You'll create a City class that bundles data and behavior together. A City object will know its own name, connections, and how to display itself. Welcome to Object-Oriented Programming.

"Testing classes is straightforward: create an object, call every method, check every attribute. If city.name returns the right name and city.is_dead_end() correctly identifies dead ends, your class works. Build the test before you build the class!"

Before You Begin
Program Architecture

Here's the structure of what you're building. city.py contains nothing but the class definition — no game code, no main(). lab-5-1.py imports it and tests every piece. Each checkpoint adds one section to the class.

# ---- city.py ----
class City:

    # --- Constructor ---
    __init__(self, name, city_type, tagline)   stores 4 private attributes
        self.__name          = name
        self.__city_type     = city_type
        self.__tagline       = tagline
        self.__connections   = {}              empty dict to start

    # --- Getter Methods ---   [CP2]
    get_name(self)                returns self.__name
    get_type(self)                returns self.__city_type
    get_tagline(self)             returns self.__tagline

    # --- Connection Methods ---   [CP3]
    add_connection(self, city_name, distance)   stores in self.__connections
    get_distance_to(self, city_name)            returns distance or -1
    get_destination_names(self)                 returns list of connected names

    # --- Status Methods ---   [CP4]
    is_win_city(self)             returns True if type == 'WIN'
    is_dead_end(self)             returns True if type == 'DEAD_END'

    # --- Display Methods ---   [CP4]
    display_destinations(self)    prints numbered menu of connections
    __str__(self)                 returns 'San Antonio (START)'


# ---- lab-5-1.py ----   [CP5]
from city import City

main():
    Create 4 City objects (San Antonio, Houston, Corpus Christi, Laredo)
    Add connections between them
    Test all getter methods
    Test __str__ by printing each city
    Test connection methods
    Test is_win_city() and is_dead_end()
    Call display_destinations()

if __name__ == '__main__': -> main()

Common Mistake Why It Fails
Forgetting self as first parameter Every method needs self first — it's how Python knows which object to work with
Using name instead of self.__name Attributes need the self. prefix to belong to the object
Naming the method _init_ (single underscores) Must be __init__ with DOUBLE underscores on both sides

Connection Methods
Add methods to manage city connections — adding routes and checking distances.
What You're Building

Three methods that let cities know about their road connections. This replaces the old CITY_DISTANCES dictionary — now each city manages its own connections!

Step-by-Step Instructions
  1. Add a comment # --- Connection Methods ---
  2. Define add_connection(self, city_name, distance) that stores the connection:
def add_connection(self, city_name, distance):
    """Add a connection to another city."""
    self.__connections[city_name] = distance
  1. Define get_distance_to(self, city_name) that returns the distance, or -1 if not connected:
def get_distance_to(self, city_name):
    """Get distance to a connected city. Returns -1 if not connected."""
    return self.__connections.get(city_name, -1)
  1. Define get_destination_names(self) that returns a list of connected city names:
def get_destination_names(self):
    """Return list of connected city names."""
    return list(self.__connections.keys())

Common Mistake Why It Fails
Using self.__connections[city_name] for get_distance_to KeyError if city not connected — use .get(city_name, -1) for safe access
Returning self.__connections.keys() directly Returns a dict_keys object, not a list — wrap with list()
Forgetting to add docstrings to methods CodeGrade checks for documentation — every method needs a docstring

What You're Building

Methods that let you ask a city about itself: "Are you a win city?" "Are you a dead end?" Plus a display method and the special __str__ method that Python calls when you print() a City object.

BookEx Patterns
  • Program 10-9 — bankaccount2.py (Step 1) — You typed a __str__ method that returned a formatted string. Python automatically calls this when you print() an object. Your City's __str__ should return something like 'San Antonio (START)'.
Step-by-Step Instructions
  1. Add a comment # --- Status Methods ---
  2. Define is_win_city(self) that returns True if self.__city_type == 'WIN' — add a docstring
  3. Define is_dead_end(self) that returns True if self.__city_type == 'DEAD_END' — add a docstring
  4. Add a comment # --- Display Methods ---
  5. Define display_destinations(self) that prints a numbered menu of connections:
def display_destinations(self):
    """Display available destinations from this city."""
    print(f'From {self.__name}, you can travel to:')
    option_num = 1
    for dest_name, distance in self.__connections.items():
        print(f'  {option_num}. {dest_name} - {distance} miles')
        option_num = option_num + 1
  1. Define __str__(self) that returns a string like 'San Antonio (START)':
def __str__(self):
    """Return string representation of the city."""
    return f'{self.__name} ({self.__city_type})'
