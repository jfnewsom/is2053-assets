The City Class
MODULE 5 • WEEK 1 • BUILD • CHAPTERS 10: CLASSES, OBJECTS, ENCAPSULATION, CONSTRUCTORS
Overview

Module 5 begins with a question: what if a city knew things about itself? Right now, city data lives in a dictionary. The city name is a key. Its connections are a nested dict. Its trivia is another dict. The functions that work on that data are scattered across your file. A class changes that. A City object holds its own name, type, tagline, connections, and trivia. The methods that work on that data live right next to it. This week you define the City class and write a test program that puts it through its paces. Nothing about the game changes yet. You are just building a better container for what already exists.

A dictionary can hold city data. A class holds city data AND knows what to do with it. That is the difference between a filing cabinet and a coworker. Both have the information. Only one can act on it.
Priya Venkatesh
Lead Architect, Bat City Collective

1
Define the City Class Skeleton
What You're Building

Create a new file called city.py. This file will contain only the City class definition, with no main() function and no if __name__ guard. The class is a module; the test program you write in CP5 is what runs it.

Step-by-Step Instructions
  1. Add your top-of-file documentation header: Name, abc123, Section, and a one-sentence description of what city.py contains.
  2. Define the class: class City:. Everything else in this file is indented inside this class.
  3. Write the class docstring immediately after the class line: a one-line description of what a City object represents.
  4. Write the __init__ method. It takes self plus three parameters: name, city_type, and tagline. Write a docstring that documents all three parameters. Store all three as private attributes using the double-underscore prefix: self.__name, self.__city_type, self.__tagline.
  5. In the same __init__, initialize two more private attributes: self.__connections = {} (an empty dict for city connections) and self.__trivia = [] (an empty list for trivia questions). These are not parameters; they always start empty.
BookEx Patterns
BookEx Patterns
  • Program 10-12 — cellphone.py — Shows exactly this pattern: class definition, __init__ constructor, and three private attributes stored with double-underscore prefix. The City class follows the same structure.
Tips & Pitfalls
Mistake What Goes Wrong Fix
Using single underscore _name instead of double __name Attribute is still accessible from outside; encapsulation is not enforced Use self.__name (two underscores) for private attributes
Forgetting self as the first parameter in __init__ TypeError: __init__ missing positional argument Every method in a class must have self as its first parameter
Putting __init__ outside the class indentation IndentationError or __init__ becomes a standalone function Every method must be indented one level inside the class

Checkpoint
2
Add Getters and __str__
What You're Building

Private attributes cannot be read from outside the class directly. Getter methods are the controlled access points. Add the getters now, then add __str__ so that printing a City object gives clean readable output.

Step-by-Step Instructions
  1. Add a comment # --- GETTER METHODS --- after the __init__ to organize the class.
  2. Write get_name(self): docstring, return self.__name.
  3. Write get_type(self): docstring, return self.__city_type.
  4. Write get_tagline(self): docstring, return self.__tagline.
  5. Write get_connection_count(self): docstring, return len(self.__connections).
  6. Write get_trivia_count(self): docstring, return len(self.__trivia).
  7. Write the __str__(self) method: return an f-string in the format 'CityName (CITY_TYPE)'. For example, a San Antonio object should print as 'San Antonio (START)'.
Why This Matters

The __str__ method is what Python calls when you use print(city_object) or embed the object in an f-string with f'{city_object}'. Without it, printing a City gives you something like <city.City object at 0x7f4a...>. With it, you get San Antonio (START). It costs one method and pays back every time you debug or display an object.

  • __str__ returns 'CityName (CITY_TYPE)' format

Checkpoint
3
Add Connection Methods
What You're Building

The __connections dictionary maps destination city names to distances. These three methods let outside code add connections and query them without touching the dictionary directly.

Step-by-Step Instructions
  1. Add a comment # --- CONNECTION METHODS --- to organize this section.
  2. Write add_connection(self, city_name, distance): docstring describing both parameters. Body: self.__connections[city_name] = distance.
  3. Write get_distance_to(self, city_name): docstring noting it returns -1 if the city is not connected. Body: use self.__connections.get(city_name, -1).
  4. Write get_destination_names(self): docstring. Body: return list(self.__connections.keys()).
  5. Write display_destinations(self): print 'From [city_name], you can travel to:', then loop through self.__connections.items() with a numbered counter and print each destination and distance.
BookEx Patterns
BookEx Patterns
  • Program 9-1 — card_dealer.py — The __connections dictionary follows the same key-value pattern as the card deck. Adding a connection is the same as adding a card. get_distance_to() uses the same .get(key, default) pattern.

Checkpoint
4
Add Trivia, Status, and Display Methods
What You're Building

Four more methods finish the class: trivia storage and retrieval, win/dead-end status checks, and a full city info display.

Step-by-Step Instructions
  1. Add # --- TRIVIA METHODS ---. Write add_trivia(self, question, answer): append a dictionary {'question': question, 'answer': answer} to self.__trivia.
  2. Write get_random_trivia(self): add import random at the top of the method (or at the top of the file). If self.__trivia is empty, return None. Otherwise return random.choice(self.__trivia).
  3. Add # --- STATUS METHODS ---. Write is_win_city(self): return self.__city_type == 'WIN'.
  4. Write is_dead_end(self): return True if self.__city_type is 'DEAD_END' or 'WRONG_WAY'.
  5. Write display_info(self): print the city name, type, tagline, connection count, and trivia count. Use the getter methods rather than accessing the private attributes directly.
Use Your Own Getters Inside the Class

Inside display_info(), call self.get_connection_count() rather than len(self.__connections) directly. Using your own methods internally is good practice. If the implementation of connection storage ever changes, only get_connection_count() needs updating.

What You Should Have

  • get_random_trivia() returns None for empty list, random dict otherwise
  • is_win_city() and is_dead_end() return the correct boolean for each city type
  • display_info() prints all five city attributes using getter methods

Write city_test.py
What You're Building

Create a second file called city_test.py. This is your test harness. It imports the City class and methodically exercises every method you just wrote. A class is only as good as its tests.

Step-by-Step Instructions
  1. Add your documentation header, then from city import City.
  2. Write a main() function. Start with a header line: print('=' * 50), print('CITY CLASS TEST'), print('=' * 50).
  3. Create four City objects: San Antonio ('START', 'Home of the Alamo'), Houston ('GATEWAY', 'Space City'), Corpus Christi ('WIN', 'Sparkling City by the Sea'), and Laredo ('DEAD_END', 'Gateway to Mexico').
  4. Test __str__: print all four objects using f-strings and verify the output format.
  5. Add connections: San Antonio to Houston (200), Corpus Christi (150), and Laredo (160). Houston to San Antonio (200). Test get_distance_to() for a connected city and for a city that is not connected. Verify you get -1 for the missing one.
  6. Add two trivia questions to San Antonio. Call get_random_trivia() and print the question and answer.
  7. Call display_info() and display_destinations() on San Antonio.
  8. Test is_win_city() and is_dead_end() on all four cities and print the results.
  9. Add if __name__ == '__main__': main() at the bottom. Run the file and verify every section produces the expected output.
