Why This Matters

Method chaining is the key idea here. When you write input(prompt).strip().upper(), Python executes left to right: input() gets a string from the user, .strip() removes whitespace from both ends of that string, and .upper() converts the result to uppercase. Each method returns a new string, and the next method is called on that result. Strings are immutable in Python, so nothing is modified in place. You are building a pipeline.

Tips & Pitfalls
Mistake What Goes Wrong Fix
Writing `input.strip().upper(prompt)` input is a function; the prompt goes inside the parentheses Write `input(prompt).strip().upper()` because call order matters
Forgetting to return the result Function runs but the caller gets None back Make sure the last expression in your function is a return statement
Using `string.upper` without parentheses You get a reference to the method object, not the result Always include the parentheses: `string.upper()`
Assigning back unnecessarily Writing `text = text.strip()` when you just want the cleaned value once If you only use it once, chain directly: `input(prompt).strip().upper()`

Refactor get_yes_no()
What You're Building

This is the payoff checkpoint. Open function_library.py and find your existing get_yes_no() function. You are going to rewrite it using .strip().upper() and watch it get shorter, cleaner, and more capable all at once.

Why This Matters

Here is what your current function has to do to handle both cases:

# BEFORE: get_yes_no() the painful way
# Four comparisons just to check two possible answers
while answer != 'y' and answer != 'Y' and answer != 'n' and answer != 'N':
    print("ERROR: Enter 'y' or 'n'.")
    answer = input(prompt)
if answer == 'y' or answer == 'Y':
    return True
else:
    return False

After normalizing with .upper(), the logic collapses. There are only two valid uppercase answers to check, and the return statement becomes a single comparison.

Step-by-Step Instructions
  1. Find get_yes_no() in your function library. Read through its current logic and count how many comparisons it makes.
  2. Rewrite the first input() call inside the function so it chains .strip() and .upper() before assigning to answer.
  3. Update the while loop condition. After normalizing, you only need to compare against 'Y' and 'N'. No need to check both cases.
  4. Update the input() call inside the loop to also use .strip().upper().
  5. Simplify the return statement. Hint: after normalization, returning whether answer equals 'Y' is all you need. No if/else required.
  6. Update the docstring to note that the function now uses .upper() for case-insensitive input handling.

Clean All Input in lab-4-2.py
What You're Building

Open lab-4-2.py from your 4.1 submission and save a copy as lab-4-2.py. Work through the entire file and apply input normalization everywhere a player types something. This is a refactor pass, not glamorous work. But it is the kind of pass that separates code that works for the developer from code that works for the player.

Step-by-Step Instructions
  1. Start at the top of your game loop. Find every input() call that reads a player choice: destination selection, menu options, hazard responses, and any other prompt that leads to a comparison.
  2. For any input that gets compared against letter strings (like 'A', 'B', 'Y', 'N'), replace the bare input(prompt) call with input(prompt).strip().upper(). You may also use your new clean_input(prompt) function from the library. Both are acceptable.
  3. For the player's name input (which is not compared against a fixed set of values), apply only .strip(). Do not convert names to uppercase.
  4. Find every while loop condition that currently checks both lowercase and uppercase (like answer != 'a' and answer != 'A') and simplify it to check only the uppercase version.
  5. Find every if/elif chain that currently checks both cases (like if choice == 'a' or choice == 'A':) and simplify each condition to check only the uppercase version.
  6. After all replacements, do a quick scan for any remaining lowercase letter literals in conditions. If you see == 'a', == 'b', or similar, those are candidates for cleanup. Verify they still make sense after normalization.
