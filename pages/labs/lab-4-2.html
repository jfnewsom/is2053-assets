Key Concepts

.upper() and .strip() compose elegantly. input(prompt).strip().upper() reads from the user, removes any surrounding whitespace, and converts the result to uppercase, all in one expression. Wrapping that pattern in clean_input() means every input cleanup in the game looks the same. .isdigit() checks whether a string consists entirely of decimal digits, which is the right test for 'is this a non-negative whole number?' before calling int(). List of dictionaries is the third dictionary shape you've seen this module: a list whose items happen to be dicts. random.choice(hazards) returns one of those dicts, and hazard['name'] retrieves a field.

Tips & Pitfalls
Forgetting the return keyword Without return, the function does the work but hands back None. The caller has nothing to compare against.
Calling .strip() after .upper() Order does not matter here in practice (both pure operations), but the canonical order is .strip().upper(): strip first, then transform.
Returning .upper() on already-uppercased data No-op cost, no correctness cost. Do not stress about double-uppering somewhere downstream.

Tips & Pitfalls
Forgetting .strip() on the input '3 '.isdigit() is False because of the trailing space. .strip() first.

Tips & Pitfalls

If Something Breaks

Common failure patterns and where they came from:

Symptom Where to Look
FileNotFoundError: hazards.txt Check CP6 (Wire It All Together). The data file must be in the same folder as lab-4-2.py. CodeGrade provides it as a fixture.
AttributeError: module 'function_library' has no attribute 'get_random_hazard' Check CP6. The hazard block in the in-game loop still calls the old function. Replace it with the new handle_hazard call.
ValueError: invalid literal for int() at a menu prompt Check CP3 (Smarter get_valid_int). The .isdigit() rewrite did not land in the library, or the wrong version of function_library.py is being used.
Yes/no prompt accepts y but rejects Y (or vice versa) Check CP4 (Smarter get_yes_no). The rewrite uses clean_input, which uppercases its input, so the comparison should be against 'Y' and 'N' only.
Trip report's HAZARDS section shows dict-shaped strings like {'name': 'Buc-ee\'s appears!', ...} Check CP6. The code is appending the entire hazard dictionary instead of just the name field. handle_hazard returns a tuple (name, escaped); unpack it and append name only.
Hazard prints fire but the trip report shows 0 hazards faced Check CP6. The counter increment moved into the wrong block, or the escaped return value is being checked for the wrong outcome.
