Build the TRIVIA Dictionary
What You're Building

Open lab-4-3.py. Near the top of the file, after your other constants, add a TRIVIA dictionary. This is the data source your trivia_gate() function will read from. Each key is a city name. Each value is a list of question dictionaries. Each question dictionary has two keys: 'question' (the text shown to the player) and 'answer' (either 'TRUE' or 'FALSE').

Named Constants

Your TRIVIA dictionary must cover at least 3 cities and include at least 2 questions per city. San Antonio should always be included as a fallback for cities that are not explicitly covered. The structure looks like this:

TRIVIA = {
    'San Antonio': [
        {
            'question': 'The Alamo was originally a Spanish mission.',
            'answer': 'TRUE'
        },
        {
            'question': 'San Antonio is the largest city in Texas by population.',
            'answer': 'FALSE'
        }
    ],
    'Austin': [
        {
            'question': 'Austin is the capital of Texas.',
            'answer': 'TRUE'
        },
        {
            'question': 'The University of Texas at Austin is located in East Austin.',
            'answer': 'FALSE'
        }
    ]
    # Add at least one more city...
}
Step-by-Step Instructions
  1. Add the TRIVIA dictionary to your constants section. Use the structure shown above as your starting point.
  2. Add entries for at least 3 cities total, including San Antonio. Cover cities your game actually visits. Check your CITY_DISTANCES dictionary for the exact city names.
  3. Write 2 questions per city at minimum. Keep questions as true/false statements about the city. Make sure every 'answer' value is exactly 'TRUE' or 'FALSE' in all caps. Your trivia_gate() function will compare against these exact strings.
  4. Add a comment above the constant: # TRIVIA (Added in 4.3): city trivia questions and answers
  5. Do a quick sanity check: for each city entry, count the items in its list and verify each has both a 'question' key and an 'answer' key.
BookEx Patterns
BookEx Patterns
  • Program 9-1 — card_dealer.py — Shows a dictionary whose values are lists, which is the same nested structure your TRIVIA constant uses. The pattern for accessing and iterating over values is identical.
Answer Values Must Be Exact

Every 'answer' value in your TRIVIA dictionary must be exactly 'TRUE' or 'FALSE' in all caps, no exceptions. Your trivia_gate() function will compare player input against these strings after calling .upper(). If your dictionary has 'True' or 'true' or True (the boolean), the comparison will silently fail.

Checkpoint
2
Write trivia_gate()
What You're Building

Open function_library.py. You will add one new function: trivia_gate(). This function takes a destination city name and the TRIVIA dictionary, picks a random question for that city, presents it to the player, and returns True if they answer correctly, False if they do not. It replaces the coin flip that has been running the Destination Gate since Module 2.

Step-by-Step Instructions
  1. Add a # TRIVIA FUNCTIONS (Added in 4.3) comment block at the bottom of your function library.
  2. Define a function named trivia_gate that accepts two parameters: destination (a string) and trivia_dict (the dictionary). Write a docstring that describes the parameters and return value.
  3. Use trivia_dict.get(destination, trivia_dict['San Antonio']) to look up the city's question list. This uses San Antonio as a fallback for any city that does not have its own entry.
  4. Use random.choice() to pick one question dictionary from the city's list. Extract the 'question' and 'answer' values into local variables.
  5. Print the trivia prompt. Show the destination city name, then the question text, then a prompt asking for TRUE or FALSE.
  6. Get the player's answer using clean_input(). The player can type T, TRUE, F, or FALSE. Normalize to 'TRUE' or 'FALSE' before comparing: if the input starts with 'T', treat it as TRUE; if it starts with 'F', treat it as FALSE. Anything else should print an error message and default to FALSE.
  7. Compare the normalized player answer to the correct answer. Print a result message: something different for correct versus incorrect. Return True if correct, False if not.
Tips & Pitfalls
Mistake What Goes Wrong Fix
Passing TRIVIA by name directly inside the function Function is tightly coupled to one global variable, which makes it harder to test Accept trivia_dict as a parameter; call trivia_gate(destination, TRIVIA) from main
Using random.randint() to pick a question Index math, off-by-one risks, breaks if list length changes Use random.choice(question_list), which is cleaner and length-independent
Comparing player input without normalization Player types 'true' and it fails; 'TRUE' works but 'True' does not Use clean_input() which already calls .strip().upper(), then compare only 'T' or 'TRUE'
Accepting only 'TRUE' or 'FALSE' with no flexibility Player types 'T' and gets penalized even though intent is clear Accept 'T' and 'TRUE' as true, 'F' and 'FALSE' as false

sorted() Works on Sets

Sets have no guaranteed order, so iterating directly can produce inconsistent output. Wrapping the set in sorted() gives you alphabetical order and makes CodeGrade's output comparison reliable: for city in sorted(unique_cities_visited):

Keep the Original Lists

visited_cities (the list) intentionally keeps duplicates; it shows the player's complete path, including revisits. unique_cities_visited (the set) strips duplicates for the count. Both serve different purposes. Do not remove the list.
