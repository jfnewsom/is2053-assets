Key Concepts

A set in Python is a collection of unique items with O(1) membership testing. my_set = set() creates an empty set. my_set.add(item) inserts an item (or does nothing if it's already present). len(my_set) gives the unique count. sorted(my_set) returns a new sorted list. Sets are unordered, so iterate them via sorted() for stable display. The trivia data shape introduced this lab is a dictionary keyed by city, with each value a list of dicts: trivia['Austin'] -> [{'question': '...', 'answer': 'TRUE'}, ...]. This combines two patterns from CP1 of Lab 4.1 (dict lookup) and CP1 of Lab 4.2 (list of dicts) into one structure.

Trivia Gate (lab-local)
What You're Building

trivia_gate(dest, trivia, current, distances) defined in lab-4-3.py: the lab file, not the library. The function looks up trivia for the destination city, picks a random question, prompts the player for TRUE or FALSE, and returns a 3-tuple (final_dest, distance, correct). A correct answer returns the chosen destination; a wrong answer calls function_library.get_redirect to detour the player elsewhere.

Why This Matters

Think about which functions belong in the library and which belong in the lab file. The library holds general-purpose helpers: data loaders, dice rolls, input cleaning, things that any future game could reuse. The lab file holds game-specific orchestration. trivia_gate is the second category: it knows the game's TRUE/FALSE answer format, the banners it prints, and the exact way it routes a wrong answer through get_redirect. None of that is reusable for the next game. Putting it in the lab file keeps the boundary clean.

"This is the heart of the new game feel. The player picks a destination, gets a question, and earns the next leg. Wrong answers are not a punishment; they are a detour, the same way get_redirect rerouted them on a TAILS flip in Module 3. Same mechanical outcome; entirely different feeling at the keyboard. Take care with the messaging on both branches. 'WRONG. The answer was X. Detour to Y instead.' reads as 'oh well, on we go.' That is the tone we want."
Sasha Mireles
Creative Director, Bat City Collective

Sets - Unique Cities
What You're Building

unique_cities = set(visited_cities) as part of per-game state initialization. unique_cities.add(current_city) every time the player arrives somewhere new, including the dead-end return-to-San-Antonio case. This replaces the post-game in-operator loop that built a deduplicated list in Lab 4.2.

Why This Matters

Lab 4.2 ended every game with a loop that scanned visited_cities and used if city not in unique_cities: to deduplicate. That works, but it's O(n) per check, O(n²) total, and the code reads like a workaround for the right data structure. set is the right data structure: uniqueness is enforced on insertion, membership is O(1), and the post-game code becomes a single len(unique_cities) call.

"Notice we are INITIALIZING the set from visited_cities, not just set(). On a fresh game the visited list is just ['San Antonio'] and the set comes out as {'San Antonio'}. On a resumed game, the loaded visited list might have several cities in it, and the set rebuilds with all of them. Both cases handled with one line. That's why set(iterable) exists."
Priya Venkatesh
Project Manager, Bat City Collective
Step-by-Step Instructions
  1. In lab-4-3.py, find the per-game state initialization block at the top of the play-again loop. It currently sets current_city, total_miles, etc.
  2. Just after the if not resumed_game: block (the one that resets the per-game counters), add: unique_cities = set(visited_cities). This runs every iteration, including the resume case, and is correct in both: a fresh game starts with a 1-element set, a resume starts with as many elements as the loaded visited_cities list had.
  3. Find every site where visited_cities.append(...) is called. There are three: arrival at a new destination, the dead-end return-to-SA branch, and (transitively) any other spot you've added cities. Mirror each append with unique_cities.add(...) right after it.
  4. Run the program. Play a multi-city game that includes a dead-end retreat (pick Laredo or Texarkana, then choose to return). Confirm the post-game Unique cities visited: count includes the dead-end city even though the player returned.
BookEx Patterns
BookEx Patterns
  • Program 9-19 — set_create.py — set(iterable) takes any iterable and builds a set from its unique elements. Same idea here: build a set from visited_cities.
  • Program 9-20 — set_add.py — my_set.add(item) inserts an item; duplicates are silently ignored. This is what makes the dead-end return-to-SA case work without extra checks.

Initializing with set() instead of set(visited_cities) Resume case breaks. A resumed game has cities in visited_cities already; the set must rebuild from them, not start empty.
Forgetting one of the .add() sites Easiest to find them all by searching for visited_cities.append. Every append site needs a mirroring add. There are usually 2 or 3.
Putting the set initialization inside if not resumed_game: Then it never runs on a resumed game, and unique_cities is undefined. The set init runs every iteration of the play-again loop; the if not resumed_game: guard is only for the counters that should reset.

What You're Building

Why This Matters

Sets are unordered. sorted(my_set) returns a new list with the elements in alphabetical order. "sorted() is a great general utility. It works on lists, on sets, on tuples, on anything iterable. It always returns a new list, so you don't have to think about whether to .copy() first. When you see the .copy().sort() pattern in your own code, that's usually a sign sorted() would be cleaner."
Danny Cavanaugh
Lead Developer, Bat City Collective
