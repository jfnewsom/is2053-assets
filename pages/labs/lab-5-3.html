Checkpoint
1
Add WinCity and DeadEndCity to city.py
What You're Building

Open city.py. First, add a get_arrival_message() method to the base City class. This is a default version that returns a generic welcome. Then add two subclasses below it: WinCity for beach destinations and DeadEndCity for dead ends. Both inherit from City and override get_arrival_message() with their own version.

Step-by-Step Instructions
  1. In the base City class, add get_arrival_message(self) at the end: return f'Welcome to {self.get_name()}! "{self.get_tagline()}"'. Use getter methods rather than self.__name directly, because subclasses cannot access the superclass's private attributes.
  2. After the closing of the City class, define class WinCity(City):. Write its __init__(self, name, tagline): the first line must be City.__init__(self, name, 'WIN', tagline) to initialize all the inherited attributes. Add any WinCity-specific attributes after that if desired.
  3. Override get_arrival_message(self) in WinCity: return a multi-line string announcing that the player has won and reached the beach. Include the city name and tagline.
  4. Define class DeadEndCity(City):. Its __init__(self, name, tagline, warning) calls City.__init__(self, name, 'DEAD_END', tagline) and stores self.__warning = warning as a private attribute.
  5. Override get_arrival_message(self) in DeadEndCity: return a string that includes the city name and the warning message.
  6. Open a Python shell and test: create a WinCity('Corpus Christi', 'Sparkling City') and a DeadEndCity('Laredo', 'Gateway', 'No beach this way!'). Call is_win_city(), is_dead_end(), and get_arrival_message() on each. Verify the right message appears for each type.
BookEx Patterns
BookEx Patterns
  • Program 11-9 — animals.py — Dog.__init__ calls Mammal.__init__(self, 'Dog'). This is exactly the pattern WinCity uses. City.__init__(self, name, 'WIN', tagline) initializes all the inherited private attributes that WinCity cannot set directly.
  • Program 11-1 — vehicles.py — Car inherits from Automobile and adds its own attribute (doors) after calling the superclass constructor. DeadEndCity does the same: it inherits from City and adds __warning after calling City.__init__.
Use Getters in get_arrival_message()

In the base City class, get_arrival_message() can use self.__name directly. But in WinCity and DeadEndCity, you cannot use self.__name. That attribute belongs to City and is name-mangled to _City__name. Use self.get_name() and self.get_tagline() in all three override versions to be safe.

Complete game.py
What You're Building

Open game.py. Add two state attributes to __init__, update __setup_cities() to use the new subclasses, then add the remaining methods that complete the game loop. This is the biggest checkpoint of the week. Work through it in the order listed.

Step-by-Step Instructions
  1. Update the imports in game.py: add from city import City, WinCity, DeadEndCity.
  2. In __init__, add two new state attributes after the existing ones: self.__game_over = False and self.__won = False. Also add self.__hazards = {} and call self.__setup_hazards() after self.__setup_cities().
  3. Update __setup_cities(): change corpus = City('Corpus Christi', 'WIN', ...) to corpus = WinCity('Corpus Christi', ...). Change Laredo and El Paso to DeadEndCity objects. The hub cities (Houston, Austin, Dallas) remain plain City objects.
  4. Write __setup_hazards(self): populate self.__hazards with the same hazard dictionary from your Module 4 game (Buc-ee's, engine light, armadillo, speed trap, etc.).
  5. Write is_game_over(self): return self.__game_over. Write did_win(self): return self.__won.
  6. Write play_turn(self): get the current City object, display it, show destinations, get the player's numeric choice using function_library.get_valid_int(), call self.__trivia_gate() with the chosen destination, call self.__travel() with the result, then call dest_city.get_arrival_message() and print it. Check dest_city.is_win_city() and dest_city.is_dead_end() to update self.__game_over and self.__won accordingly.
  7. Write __trivia_gate(self, chosen_dest, distance, all_destinations): call dest_city.get_random_trivia(). If None, auto-pass. Otherwise present the question, get the player's answer using function_library.clean_input(), compare, record the result on the player with self.__player.record_trivia(correct), and return (final_destination, final_distance).
  8. Write __travel(self, destination, distance): run the D20 loop using function_library.roll_d20(), call self.__player.add_turn() and self.__player.add_miles() each iteration, check for hazards with function_library.check_percentage(30), and when the remaining distance hits zero call self.__player.travel_to(destination, distance).
  9. Write __handle_hazard(self): pick a random hazard key, flip a coin with function_library.flip_coin(), print the result, and call self.__player.record_hazard(hazard_name, escaped).
  10. Write display_final_report(self): print the player's name, route string, unique cities (sorted), trivia stats with accuracy percentage, hazard stats, total miles and turns, and win/loss status.
Polymorphism in play_turn()

When you call dest_city.get_arrival_message() in play_turn(), Python automatically calls the right version. If dest_city is a WinCity, it calls WinCity.get_arrival_message(). If it is a plain City, it calls City.get_arrival_message(). The Game class does not need to check the type. The objects handle it themselves.
