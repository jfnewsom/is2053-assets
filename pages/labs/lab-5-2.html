One class is a container. Multiple classes working together is architecture. This week you add two more pieces to your program. First, the Player class: an object that knows where the player is, how far they have traveled, and what cities they have visited. Second, a Game class skeleton that holds the cities dictionary and the player, and can start a new game. You will wire them together in journey_test.py, which simulates a simple trip without a full game loop. Next week you complete the Game class and add inheritance. This week is about getting the objects talking to each other.

When classes work together, each one should do its own job and nothing else. The Player tracks the player. The City knows about the city. The Game coordinates between them. If you find yourself putting travel logic in the City class or city setup logic in the Player class, something is off.

Write the Player Class
What You're Building

Create player.py. The Player class tracks everything about the player's journey: where they are, how far they have gone, how many turns have elapsed, and where they have been. All of this used to be scattered as global variables in your game loop. Now it lives in one object.

Step-by-Step Instructions
  1. Add a documentation header. Define class Player: with a class docstring.
  2. Write __init__(self, name, starting_city='San Antonio'). Initialize private attributes: self.__name, self.__current_city (set to starting_city), self.__total_miles = 0, self.__total_turns = 0. Also initialize self.__visited_cities = [starting_city] (a list, preserves duplicates) and self.__unique_cities = set() then immediately self.__unique_cities.add(starting_city).
  3. Write getter methods: get_name(), get_current_city(), get_total_miles(), get_total_turns(), get_visited_cities() (return a copy with .copy()), and get_unique_count() (return len(self.__unique_cities)).
  4. Write travel_to(self, city_name, distance): update self.__current_city, add distance to self.__total_miles, append city_name to self.__visited_cities, and call self.__unique_cities.add(city_name).
  5. Write add_turn(self): increment self.__total_turns by 1.
  6. Write add_miles(self, miles): add miles to self.__total_miles.
  7. Write get_route_string(self): return ' -> '.join(self.__visited_cities).
  8. Write __str__(self): return f'{self.__name} at {self.__current_city} ({self.__total_miles} mi)'.
BookEx Patterns
BookEx Patterns
  • Program 10-9 — bankaccount2.py — The deposit() method adds to __balance using +=. travel_to() does the same thing: it updates __total_miles and __current_city just like deposit() updates __balance. The pattern is identical.

  • get_visited_cities() returns a copy, not the original list
  • __str__ returns 'Name at City (X mi)' format
Checkpoint
2
Write the Game Class Skeleton
What You're Building

Create game.py. The Game class is the orchestrator. It holds all the City objects and the Player, and it will eventually run the full game loop. This week you build the foundation: city setup and the ability to start a new game. The game loop itself comes in 5.3.

Step-by-Step Instructions
  1. Add imports at the top: import random, import function_library, from city import City, from player import Player.
  2. Define class Game: with a class docstring. Write __init__(self): initialize self.__cities = {}, self.__player = None, then call self.__setup_cities().
  3. Write __setup_cities(self) as a private method (double underscore). Inside, create City objects for at least 5 cities: San Antonio (START), Houston (GATEWAY), Austin (CORRIDOR), Corpus Christi (WIN), and Laredo (DEAD_END). Use the taglines from your Module 4 game.
  4. After creating the City objects, add connections between them using city_object.add_connection(name, distance). Use the same distances from your Module 4 game. Add at least 3 trivia questions to San Antonio and Houston using city_object.add_trivia(question, answer).
  5. Store each city in self.__cities with the city name as the key: self.__cities['San Antonio'] = san_antonio.
Private Methods Use Double Underscore Too

Just like private attributes, private methods use the double-underscore prefix: def __setup_cities(self):. This prevents outside code from calling setup directly. The method is called once from __init__ and never needs to be called again.

3
Wire Player into Game
What You're Building

Add two more methods to the Game class: start_new_game(), which creates the Player object, and get_current_city(), which returns the City object for wherever the player currently is. These are the two methods journey_test.py will use.

Step-by-Step Instructions
  1. Write start_new_game(self): get the player's name with input() and .strip(). Create self.__player = Player(name, 'San Antonio'). Print a welcome message.
  2. Write get_current_city(self): get the player's current city name with self.__player.get_current_city(), then return self.__cities[city_name]. This returns the actual City object, not just the name.
  3. Write get_player(self): return self.__player. This lets journey_test.py access the Player object to call travel methods.
get_current_city() returns a City object, not a string. That is the key difference from Module 4. Instead of looking up a name in a dictionary, you are handing back something that knows what to do with itself. The Game does not have to know how to display destinations. It just asks the City to do it.

Write journey_test.py
What You're Building

Create journey_test.py. This simulates a short trip using your Game and Player classes together. There is no game loop yet. You will manually call travel steps to verify everything is wired up correctly.

Step-by-Step Instructions
  1. Add imports: from game import Game.
  2. Write main(). Create a Game object. Call game.start_new_game().
  3. Get the current city object: city = game.get_current_city(). Print it and call city.display_destinations().
  4. Simulate travel: call game.get_player().travel_to('Houston', 200). Print the player object to verify the updated state.
  5. Simulate two more travel steps to different cities. After each one, print the player and verify the city name and mileage update.
  6. After all three steps, print the player's full route using game.get_player().get_route_string() and the unique city count using game.get_player().get_unique_count().
  7. Add the if __name__ == '__main__': guard and call main().
Hardcoded Travel in the Test Is Fine

In journey_test.py, calling game.get_player().travel_to('Houston', 200) directly is correct. This is a test program. You are verifying the objects work, not simulating real gameplay. The real game loop with user input comes in 5.3.
