MODULE 3 • WEEK 1 • BUILD • CHAPTERS 6: FILES AND EXCEPTIONS
Overview

You've built an amazing road trip game over Module 2 — full validation, multiple legs, session statistics across multiple games. But there's one huge problem: everything vanishes when you quit! Imagine playing for 20 minutes, racking up three games of session stats, then accidentally closing the program. All that progress — gone. This week, you'll learn to save your full session to a file and load it back when you return. Welcome to file I/O — the skill that makes data persist!

"Module 2 was about logic. Module 3 is about persistence. Once your program can save state to disk and load it back, you cross a line. You stop writing scripts and start writing real software."

What You're Building

A function that reads the 9 lines you wrote in CP2, converts them back to the right types, and returns them as a tuple. If the save file doesn't exist, it returns None instead of crashing.

Why This Matters

Reading from disk has two failure modes: the file might not exist (player has never saved), or the data might be the wrong type (integers come back as strings until you int() them). The try-except pattern handles the first; the conversion handles the second. Without both, your program crashes the first time anyone clicks Load.

Resume from Saved
What You're Building

The Load branch from CP4. When the player chooses 2 at the main menu, you call load_game(), check the return value, and either restore the 9 fields or fall back to a new game. You also set a resumed_game flag so the play-again loop knows not to reset per-game state on the first iteration.

Why This Matters

Unpacking a 9-tuple cleanly is a real skill. Get the order wrong and your current_city becomes 350. The resumed_game flag is the trickier part: your play-again loop resets per-game state at the top of each iteration. When you resume a saved mid-trip game, you want to KEEP the loaded per-game state, not reset it. One Boolean flag, used once on the first iteration only, threads the difference through cleanly.

Step-by-Step Instructions
  1. Before the play-again loop, add resumed_game = False. This will be the default for New Game
  2. In the if main_choice == 2: branch, call saved = load_game()
  3. If saved is None: print 'No saved game found. Starting new game.' and set main_choice = 1 so the New Game branch runs next
  4. Otherwise (saved is a tuple): unpack the 9 values into the same variable names you used in save_game(). Use parallel assignment: (player_name, current_city, total_miles, total_turns, total_hazards, total_escaped, games_played, session_miles, session_hazards) = saved
  5. Print a welcome-back message using the loaded values
  6. Set resumed_game = True so the play-again loop knows not to reset
  7. Inside the play-again loop, change the per-game reset to be conditional: if not resumed_game: then reset, otherwise skip the reset
  8. Set resumed_game = False at the end of the conditional, so only the FIRST iteration resumes from save; subsequent iterations reset normally

Mid-Trip Save Option
What You're Building

Replace the existing 'Keep going this trip? (y/n): ' prompt with a 3-way choice: 1. Continue journey, 2. Save and quit, 3. Quit without saving. Save & quit calls save_game() and exits all the way back to the main menu, skipping the trip summary.

Why This Matters

Save & quit is not the same as 'finish the game.' When the player saves mid-trip, the game is paused, not over. They don't want to see a TRIP SUMMARY or SESSION STATISTICS that suggests the game is done; they want to go back to the main menu so they can quit or pick up later. The saved_and_quit flag is how you skip the end-of-game displays cleanly.

"Real talk: I shipped a save feature on a side project once where 'Save and quit' fell through to the trip summary. Players saw 'Status: Trip Incomplete' even though they had just saved a perfectly fine game in progress. They thought the save had failed. I had to patch it within a week. The fix is exactly what we are doing here: a flag that says 'this was a save, not an end-of-game,' and we honor it in the post-loop code. Two extra lines, no confused players."

Junior Developer, Bat City Collective

Tips & Pitfalls
Save & quit shows trip summary anyway You forgot the if saved_and_quit: guard after the in-game loop, OR you forgot to continue back to the main menu (instead falling through to the trip summary).
Quit without saving acts the same as Save & quit The quit_without_saving branch should NOT call save_game() and SHOULD print the trip summary. The two branches are different paths.
Forgetting to pass all 9 args to save_game() Python error missing N required positional arguments. The call site is long — line it up carefully and verify all 9 are present.
Wrapping the wrong block in if not saved_and_quit: The TRIP SUMMARY and SESSION STATISTICS are both gated, but at different levels: TRIP SUMMARY is gated inside the play-again loop, SESSION STATISTICS is gated after the play-again loop ends.

If Something Breaks

Common failure patterns and where they came from:

Symptom Where to Look
FileNotFoundError when starting a New Game Check CP3 (load_game) — the except FileNotFoundError clause is missing or in the wrong place. Starting a New Game should NOT call load_game(), so this also means the main menu branch in CP4 is calling load_game() for choice 1 by mistake.
After choosing Load, game starts at San Antonio with miles = 0 Check CP5 (Resume) — either the tuple unpacking failed, or the resumed_game flag isn't preventing the per-game reset at the top of the play-again loop. Add a print statement right after unpacking to confirm the loaded values are correct.
Save & quit prints a TRIP SUMMARY even though the game wasn't over Check CP6 (Mid-Trip Save Option) — the if saved_and_quit: guard after the in-game loop is missing or wraps the wrong code. The TRIP SUMMARY print should be inside the else branch.
All saved values come back as strings (game crashes with TypeError later) Check CP3 (load_game) — the 7 numeric fields need int() wrapping. Only player_name and current_city stay as strings.
savegame.txt contains all 9 values on a single line Check CP2 (save_game) — you forgot the + '\n' on each write() call. Each field needs its own newline.
Main menu doesn't reappear after a game finishes Check CP4 (Main Menu) — the entire game logic must be indented INSIDE the while in_program: loop. If a closing brace got misplaced, the program will run the game once and exit.
