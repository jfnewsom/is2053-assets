Replace Hazard if/elif with HAZARD_NAMES List
What You're Building

Your hazard selection currently uses a chain of if/elif branches or a series of random number comparisons. You are going to apply the same parallel-list pattern from CP2 last week: define HAZARD_NAMES as a list constant, then write a get_random_hazard() function that picks a random index and returns the name at that position.

Step-by-Step Instructions
  1. Start by copying lab-3-2.py to a new file named lab-3-3.py. Also make a copy of your current function_library.py — both files get updated this week.
  2. Near the top of lab-3-3.py, add the HAZARD_NAMES list constant with at least six hazard strings. Include the hazards already used in your game, plus any new ones you want to add.
  3. Add the REPORT_FILE = 'trip_report.txt' constant nearby.
  4. Define a get_random_hazard() function. It should use random.randint() to generate a random index in the range 0 to len(HAZARD_NAMES) - 1, then return HAZARD_NAMES[index].
  5. In the game loop, find where hazards are triggered. Replace the old if/elif hazard selection logic with a call to get_random_hazard(). The return value is the hazard name string — use it the same way you used the old hazard variable.
  6. Run the game and trigger a few hazards to confirm different hazard names appear and they all come from your list.
Generating a valid random index

To pick a random position in a list, generate a number from 0 up to — but not including — the list length: index = random.randint(0, len(HAZARD_NAMES) - 1). Then use HAZARD_NAMES[index] to get the hazard name at that position.

Checkpoint
2
Add Sorted Hazard Display and Unique Cities Calculation
What You're Building

This checkpoint adds two list operations to the final trip report section. First, you will sort the hazard history alphabetically so it reads cleanly — but you must copy the list before sorting so the original order is preserved. Second, you will calculate how many unique cities the player visited by filtering out duplicates using the in operator.

Step-by-Step Instructions
  1. In the final trip report section of your game, add the sorted hazard display. First, make a copy of the list: sorted_hazards = hazards_list.copy(). Then sort the copy in place: sorted_hazards.sort(). Loop through sorted_hazards to display them — not hazards_list.
  2. Add the unique cities calculation. Create an empty list unique_cities = []. Loop through visited_cities. For each city, check if city not in unique_cities — if it is not already there, append it.
  3. Use len(unique_cities) in the trip report to display the count of unique cities visited.
  4. Run the game, visit a city more than once, and confirm the unique count is lower than the total route length.
Always copy before sorting

sort() modifies the list in place permanently. If you call hazards_list.sort() directly, the original encounter order is gone — which breaks the save file if the player saves again. Always create a copy first: sorted_hazards = hazards_list.copy(), then sort the copy.

Write save_trip_report() and Generate trip_report.txt
What You're Building

The trip report currently only prints to the screen. This checkpoint adds a save_trip_report() function that writes a formatted version to trip_report.txt so the player can keep a record of their run.

Step-by-Step Instructions
  1. Define save_trip_report() with the following parameters: player_name, current_city, total_miles, total_turns, hazards_faced, hazards_escaped, visited_cities, and hazards_list. Give it a docstring.
  2. Inside the function, open REPORT_FILE in write mode using the with statement.
  3. Write the header section: a title line, a separator line of equal signs, the driver name, and the final destination.
  4. Write a ROUTE section. Loop through visited_cities and write each city separated by ' -> ' on a single line. Then write the unique cities count on the next line.
  5. Write a STATISTICS section with total miles, total turns, hazards faced, hazards escaped, and the escape rate as a percentage (use :.1f formatting). Guard against division by zero on the escape rate.
  6. Write a HAZARDS (A-Z) section. Copy and sort hazards_list inside the function, then write each hazard prefixed with ' - '.
  7. Write a closing separator line.
  8. In the main game, after the trip report prints to screen, ask the player if they want to save the report using function_library.get_yes_no(). If yes, call save_trip_report() with all the required arguments.

Scope Reminder

Chapter 7 list methods are all in scope: append(), sort(), copy(), remove(), index(), and len(). The with statement remains permitted for file operations. Do NOT use sets, dictionaries, classes, list comprehensions, or any Chapter 8+ features. The set() type is the Module 4 upgrade for unique tracking — use the for/in approach this week.

Permitted Techniques

Chapters in scope: 2–7 (all techniques from Chapters 2–6 plus full Chapter 7 list processing)

  • list.copy() to duplicate a list before sorting
  • list.sort() to sort a list in place
  • for loop with range(len(list)) for indexed display
  • in operator to check membership and build unique lists
  • with open(filename, 'w') for file writing
  • random.randint(0, len(list) - 1) for random index generation

Not permitted: set(), dict, classes, list comprehensions, sorted() built-in (use .sort() on a copy instead)
