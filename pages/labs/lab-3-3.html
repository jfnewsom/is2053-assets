"Last week you saw what parallel lists do for destinations. This week, you do the same trick for hazards — but with three lists per item instead of two. What You're Building

Three parallel lists in lab-3-3.py: HAZARD_NAMES, HAZARD_HEADS_MSGS, and HAZARD_TAILS_MSGS. Each list has six items, and items at the same index describe the same hazard. Same idea as the destination lists from Lab 3.2: same index across three lists = related data. With six hazards instead of three, the savings are bigger. Module 4 takes this to its conclusion by moving the data out of code entirely — into a text file.

Sort Hazards Alphabetically in the Trip Summary
"The player just survived a road trip. Their brain is still in the chaos of it. The trip summary is where we give them order. A sorted list of hazards reads differently than a chronological one — alphabetical feels like a record, like an index, like finalized. Two lines of code, big UX win."
Sasha Mireles
Creative Director, Bat City Collective
What You're Building

A sorted copy of hazards_list that gets displayed alphabetically in the trip summary. The original hazards_list stays in arrival order — save_game needs the chronological version, and the player's trip narrative matches arrival order too. So we copy first, sort the copy, display the copy. The original is untouched.

Why This Matters

list.sort() is a destructive operation — it changes the list in place. If you call hazards_list.sort() directly, you've permanently lost the arrival order. .copy() creates an independent duplicate; sort the duplicate and the original stays intact. This is the same .copy() pattern you saw in get_redirect in Lab 3.2 — preserve the source, mutate the copy.

Step-by-Step Instructions
  1. Find the trip summary block in main() — the part that prints YOUR ROUTE and HAZARDS ENCOUNTERED. It comes after the in-game loop ends and before the play-again prompt
  2. Immediately before the HAZARDS ENCOUNTERED display, add two lines: sorted_hazards = hazards_list.copy() and sorted_hazards.sort()
  3. Change the display call from displaying hazards_list to displaying sorted_hazards
  4. Update the section title from 'HAZARDS ENCOUNTERED' to 'HAZARDS ENCOUNTERED (A-Z)' so the sort order is visible to the player
  5. Important: only the display uses sorted_hazards. The save_game call (if it fires) still passes the original hazards_list in arrival order. Don't touch that path
BookEx Patterns

Section 7.5.4 (list.sort()): sorts a list in place alphabetically by default. Section 7.5.6 (list.copy()): creates an independent copy so the source list is preserved. Together they're the standard 'sort-without-mutating' pattern in Python.

What You're Building

A second list, unique_cities, built from visited_cities by filtering out duplicates with the in operator. The count of unique cities goes in the trip summary right after the route block, and the full list goes into the trip report file in CP7.

Why This Matters

The in operator is the cleanest way to check whether something already exists in a list. The pattern is: empty list, loop through the source, append to the new list only if the item isn't there yet. Four lines. This is what Python's set() type does internally — you'll learn that in Module 4. For now, the explicit pattern is the learning.

Step-by-Step Instructions
  1. In main()'s trip summary block, right after the sorted_hazards lines from CP5 (or just before, doesn't matter), add the unique-cities pattern
  2. Initialize an empty list: unique_cities = []
  3. Loop through visited_cities: for city in visited_cities:
  4. Inside the loop, check membership: if city not in unique_cities: then unique_cities.append(city)
  5. Display the count right after the route display: print(f'Unique cities visited: {len(unique_cities)}') and then print()
